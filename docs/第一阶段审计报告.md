  ────────────────────────────────────────
  层级: 生效层
  定义: Claude Code 会实际读取并用于决策的内容。AI 收到后会据此约束自身行为，但没有硬性拦截机制，理论上 AI 可以违反
  特征: 被 Claude Code 解析引擎加载；实际影响 AI 行为；但无 exit 2 级别的强制力

  层级: 说明层
  定义: 人工阅读的文档，Claude Code 不会主动加载，除非 AI 被明确指示去读取
  特征: 不影响 AI 行为，除非人工指示读取；主要服务于人工审查

  现状

  5 个 commands 文件（spec.md / design.md / tasks.md / implement.md / verify.md）仍引用旧的 specs/<功能名>/ 路径和

  3.5 自动执行闭环总体方案

  第一阶段的闭环是最小闭环：

  用户输入需求
    → /spec 生成 goal.md + spec.md + feature.json(draft)
    → 人工审查 → status: review → 人工批准 → status: approved
    → /design 生成 plan.md + design.md
    → /tasks 生成 steps.md + tasks.md
    → 人工确认 → phase_gate_approved: true
    → /implement 逐条执行（gate-write 硬拦截非法写入）
    → /verify 运行 required_checks（gate-bash 白名单模式）
    → 检查双文档一致性
    → status: done（或 blocked 等待人工）

  自动化边界：
  - 自动的：hooks 拦截（gate-write、gate-bash）
  - 半自动的：/verify 中的 required_checks 执行和双文档对比
  - 手动的：状态推进（draft→review→approved）、阶段闸门批准

  3.7 如何确保第一阶段完成前不误入第二阶段

  1. feature.json 中的 phase 字段：第一阶段默认 phase: "phase1"
  2. phase_gate_approved 字段：默认 false，只有人工确认后才能设为 true
  3. governance-core.md 规则：明确写"第一阶段完成后不得自动进入第二阶段"
  4. 验收清单.md 的负向检查：第 F 项"确认未进入第二阶段"
  5. 会话约束：Claude 不得仅凭会话口头确认进入第二阶段，必须有 feature.json 中的 phase_gate_approved = true
  且批准信息完整

  5.2 还不能进入第二阶段的真实原因

  逐项列出：

  1. 5 个 rules 文件缺少"执行层级标注"：用户无法分辨哪些约束是硬性拦截的、哪些靠 AI 自觉——存在虚假安全感风险
  2. 验收清单第 8 项描述错误：将 hooks 注册位置写成了项目级 .claude/settings.json，实际在用户级
  ~/.claude/settings.json——验收时会产生困惑
  3. commands 迁移方案未落地：5 个 commands 文件仍含错误的 specs/ 路径和 /project:* 引用——如果 AI 触发了 command 而非
  skill，会执行错误路径
  4. agents 兼容警告未追加：designer.md 和 spec-writer.md 仍指向旧路径，无任何提示——可能导致新用户误用
  5. gate-bash.js 的 fail-open 决策未在任何文档中标注：用户不知道 Bash hook 异常时是放行而非拦截
  6. 验收清单本身未被逐项勾选验证：清单存在但所有项均为 [ ]，需有人实际验证并勾选

  > **gate-bash.js 失败策略**：fail-open（异常时放行）。
  > **设计理由**：对 Bash 采用 fail-closed 会导致 JSON
  解析异常时所有命令被误拦，严重影响基本开发体验。这是安全性与可用性的折中。gate-write.js 采用 fail-closed
  是因为写入操作的风险更高，宁可误拦也不放行。

  非程序员验收说明

  如果你不是程序员，以下是你需要知道的：

  这套体系在保护什么？

  它防止 AI 在未经你批准的情况下修改你的项目代码。每个新功能必须经过"提出→审查→批准→实现→验收"的完整流程。你在每个环节都
  有审查和否决权。

  你需要看懂的是"小白版"文档

  每个功能会产生 4 份"小白版"文档（用日常语言写的），和 4 份"AI 执行版"文档（技术语言写的）。你只需要看小白版：

  - goal.md：这个功能要做什么、允许做什么、不允许做什么
  - plan.md：打算怎么做、会改哪里、不会改哪里
  - steps.md：执行步骤清单（可以看进度）
  - acceptance.md：验收标准和验收结果

  你的审查权力

  - AI 不能自己批准功能——必须你说"可以做"才能开始写代码
  - AI 不能自己决定进入下一阶段——必须你确认
  - 如果 AI 试图修改你没批准的文件，系统会自动拦截（这是"硬拦截"，AI 无法绕过）
