> 在执行到相关任务点时，请主动参考本文档中的对应章节。

## 难点 1：PreToolUse 门禁脚本的路径反推

### 对应需求章节
- 第十节：批准状态的进入条件（14 项前置检查）
- 第十一节：范围控制
- 第十二节 2：PreToolUse(Edit|MultiEdit|Write) hook

### 问题描述

PreToolUse hook 收到的是 Claude 即将写入的目标文件路径，该路径并不在 features/ 目录下。hook 脚本需要从这个路径反推出对应的 feature，找到 features/<feature-id>/feature.json，然后检查 14 项前置条件。

### 解决方案

**约束：任何时刻只允许一个 feature 处于 implementing 状态。**

gate-write.js 扫描 features/ 下所有 feature.json，找到 status === "implementing" 或 "verifying" 的唯一 feature。然后读取其 allowed_paths / forbidden_paths，用 glob 匹配判断目标路径是否在允许范围内。

### 边界情况

- 没有活跃 feature -> 阻断所有业务代码写入
- 多个活跃 feature -> 阻断并报错"状态异常"
- 写入 features/*/ 下的文档文件 -> 放行（spec 文档编辑不受门禁限制）
- 写入 .claude/ 目录 -> 放行（治理体系自身配置）

### 路径匹配规则（补充硬约束）

- 必须使用项目根相对路径
- 必须使用统一 glob 语法（minimatch 风格）
- 路径匹配前先做 Windows 分隔符归一化（反斜杠转正斜杠）
- forbidden_paths 永远优先于 allowed_paths

## 难点 2：Stop hook 的阻断计数持久化

### 对应需求章节
- 第十二节 6：Stop / SubagentStop
- 第十二节 7：Stop hook 逃生口

### 问题描述

Stop hook 每次被调用是独立进程，无法在内存中维持"已阻断次数"。

### 解决方案

在活跃 feature 的 evidence/ 目录中维护 .stop-block-count 文件。

逻辑流程：
1. Stop hook 触发时读取计数
2. override_stop_guard === true -> 直接放行
3. 计数 < max_stop_blocks -> 阻断（exit 2），计数 +1
4. 计数 >= max_stop_blocks -> 放行，记录逃生日志，状态改为 blocked
5. 新的 implement 轮次开始时重置计数

## 难点 3：状态机流转合法性校验

### 对应需求章节
- 第九节 3：状态机完整约束
- 第九节 4：状态推进职责

### 解决方案

gate-write.js 中维护合法转换表：

| 当前状态 | 允许目标 | 推进者 |
|----------|----------|--------|
| draft | review | Claude（文档齐备后） |
| review | approved | **仅人工** |
| approved | implementing | Claude（前置条件满足后） |
| implementing | verifying | Claude（实现完成后） |
| verifying | done / blocked | Claude（检查全通过 / 达到停止条件） |

每次写入 feature.json 时校验新旧状态转换合法性。
review -> approved 需额外检查 approved_by、approved_at、approval_method 三项非空。

## 难点 4：双文档一致性检查

### 对应需求章节
- 第五节 3：两套规范之间的关系

### 解决方案

在 /verify skill 中增加结构化对比步骤：
1. 读取 goal.md 的目标和范围段落
2. 读取 spec.md 的概述和功能需求段落
3. 对比：功能目标、允许/禁止范围、验收标准口径
4. 输出对比表到 verify.md
5. 有实质分叉 -> 验收不通过

### 局限性

此检查是 best-effort（依赖 LLM 语义判断），定位为"辅助发现分叉的工具"而非"杜绝分叉的门禁"。

## 难点 5：PreToolUse(Bash) 的双层策略

### 对应需求章节
- 第十二节 3：PreToolUse(Bash)
- 第十三节 2：Bash 执行控制策略

### 解决方案

通过 feature.json 的 status 字段区分：
- status === "verifying" -> 白名单模式（只允许 required_checks 中的命令）
- 其他状态 -> 黑名单模式（拦截高危命令）

黑名单覆盖 Windows + bash 双平台高危命令。
白名单使用前缀匹配，普通探索性命令不受限制。

### 平台权威性（补充硬约束）

- command_windows 为必填且权威字段
- command_bash 只作为可选兼容字段
- 若填写 command_bash，必须与 command_windows 语义等价

## 难点 6：Windows 环境下 hook 脚本的可靠执行

### 对应需求章节
- 第八节 7：Windows shell 策略
- 第十二节：全部 hooks

### 解决方案

**所有 hook 脚本统一使用 node .claude/hooks/xxx.js 执行**

关键原则：
- **异常时放行**：hook 自身出错时 exit 0，避免 bug 锁死开发
- **路径标准化**：比对前用 path.resolve() 标准化
- **JSON 零依赖**：feature.json 用 JSON，Node.js 原生支持
- hook 通过 stdin 接收 JSON（含 tool_name 和 tool_input）
- 阻断时 exit 2，放行时 exit 0

### hook 须覆盖的工具类型（补充硬约束）

PreToolUse 和 PostToolUse 的 if 条件必须写为：
- Edit || MultiEdit || Write（不能遗漏 MultiEdit）
- 否则 Claude 可通过批量编辑绕过门禁
