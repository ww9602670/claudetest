# 面向编程小白的 Claude Code Spec 治理体系需求（终极版）

## 一、任务性质

我要你为当前项目设计并落地一套“适合编程小白管理 AI 开发”的 spec 治理体系，并且这套体系必须能在 Claude Code 中真正被加载、真正被执行、真正支持自动验证、自动修复、自动迭代。

这不是普通文档整理任务，而是一个“小白可审 + AI 可执行 + Claude Code 可落地 + 自动闭环”的开发治理体系设计与重构任务。

---

## 二、背景与目标

### 1. 背景

我能看懂目标和结果，但看不懂传统 spec 里过于专业的 design 和 tasks。  
我不希望牺牲开发质量，也不希望 AI 因为规范太松而乱做。

所以我要的不是把规范简化成空话，而是把规范改造成：

- 我能审核
- AI 能执行
- 结果能验收
- 流程能自动化
- 边界能被约束
- 证据能被沉淀

### 2. 总目标

请你设计一套新的 spec 治理体系，替代当前“传统 spec 过于专业、命令与规则落位不一致、自动执行缺少硬门禁”的状态。

这套体系必须同时满足：

1. 非程序员可以审查和把控
2. AI 可以按规范执行开发
3. 保留开发质量门槛
4. 支持逐步验收
5. 支持长期复用到其他项目
6. 支持 Claude Code 低上下文成本加载
7. 支持在 Claude Code 中自动执行、自动验证、自动修复、自动迭代
8. 支持 Windows 环境稳定落地

---

## 三、对当前项目现状的判断

当前项目中，Claude 已创建出一套初版结构，包括：

- `CLAUDE.md`
- `.claude/`
  - `agents/`
  - `commands/`
  - `rules/`
- `specs/`
  - `README.md`

这套结构目前只能算“初版脚手架”，还不能视为完整治理体系。

### 当前已发现的主要问题

1. **命令引用与命令结构不一致**
   - 当前 `.claude/commands/` 下是扁平命令文件：
     - `spec.md`
     - `design.md`
     - `tasks.md`
     - `implement.md`
     - `verify.md`
   - 但这些命令正文内部却引用了 `/project:design`、`/project:tasks`、`/project:implement`、`/project:verify` 等调用形式。
   - 这会导致“命令文件如何落地”和“命令文案如何调用”之间存在不一致风险。

2. **当前规则只约束 `specs/**`**
   - 现有 `.claude/rules/spec-workflow.md` 只对 `specs/**` 生效。
   - 它可以做文档互锁，但不能作为全局实现门禁，无法可靠阻止 Claude 直接去改业务源码。

3. **当前 spec 体系仍然是传统技术三件套主导**
   - 当前命令与 agents 设计围绕：
     - `spec.md`
     - `design.md`
     - `tasks.md`
     - `verify.md`
   - 这套体系对 AI 执行是熟悉的，但对编程小白并不友好。

4. **当前尚未形成自动闭环**
   - 还缺少：
     - hooks 级别的硬门禁
     - 机器可读状态文件
     - 自动验证与自动修复边界
     - 停止条件
     - 失败升级机制
     - 统一证据链目录规范

因此，本次任务不是在现状上“轻微补字”，而是要在保留可复用资产的前提下，**重构为真正可执行的治理体系**。

## 四、必须明确的总体原则

### 1. 说明层与生效层必须严格区分

后续体系必须分成两层：

#### A. 说明层
给人审阅、理解、审核使用。

建议放在：

- `docs/spec-system/`
- `docs/需求完整版.md`
- `docs/spec-system/模板/`
- `docs/spec-system/示例/`

#### B. 生效层
给 Claude Code 真正读取和执行使用。

必须落在：

- `CLAUDE.md`
- `.claude/settings.json`
- `.claude/rules/*.md`
- `.claude/skills/**/SKILL.md`
- `.claude/commands/*.md`（仅兼容保留）
- `.claude/agents/*.md`
- `.claude/hooks/*`
- 功能目录下的机器可读状态文件

要求：

**任何只写在 `docs/` 里的内容，都不能默认视为 Claude Code 已经会执行。**  
必须明确说明它通过什么路径进入执行体系。

### 2. 单一执行态目录根必须明确

本项目**只能有一个执行态功能目录根**。

- `features/<feature-id>/` 是唯一执行态目录根
- `specs/` 仅允许保留：
  - 历史迁移说明
  - 模板样例
  - `README.md`
  - 迁移期兼容文档
- `specs/` 不再作为新功能运行入口
- 任何新功能不得在 `specs/` 下创建执行态文档
- hooks、rules、agents、skills、commands 对功能文档的读取与写入，均默认以 `features/<feature-id>/` 为执行态路径

### 3. 单一事实源原则

治理体系中的以下关键信息必须只认仓库内显式文件，不得依赖 Claude 的 auto memory、临时会话记忆或口头说明：

- 批准状态
- 当前阶段
- 允许修改范围
- 禁止修改范围
- 停止条件
- 人工接管条件
- 自动修复轮次
- 当前 feature 的真实执行根路径

单一事实源仅限于：

- `features/<feature-id>/feature.json`
- 小白版文档
- AI 执行版文档
- 项目根规则文件
- 说明层中被明确指定为权威的映射说明

## 五、必须拆分为两套规范产物

这是本次终极版最重要的修改点：

**小白版和 AI 执行版不能混写，必须分开。**

### 1. 小白版（人类审核主入口）

这是给非程序员审核、把控、验收用的主规范。

每个功能目录必须包含以下 4 份文档：

1. `goal.md`：目标说明
2. `plan.md`：方案翻译
3. `steps.md`：执行清单
4. `acceptance.md`：验收说明

这 4 份文档是**人类审核主入口**。

要求：

- 必须优先保证非程序员看得懂
- 必须能用于范围审核
- 必须能用于逐步验收
- 必须能发现偏题、漏项、越界
- 不得退化成空话

### 2. AI 执行版（内部技术执行规范）

这套保留传统 spec 方式命名，供 AI 实现、验证、审查使用。

每个功能目录必须包含：

1. `spec.md`
2. `design.md`
3. `tasks.md`
4. `verify.md`

这 4 份文档是**AI 执行版技术规范**。

要求：

- 继续沿用 spec 方式命名
- 服务于 AI 的实现、测试、审查、验证
- 不再作为非程序员的主审核入口
- 必须与小白版一一映射，不允许两套体系长期并行却互相脱节

### 3. 两套规范之间的关系

> ⚠️ **技术难点参考**：双文档一致性检查的实现局限性，请参考 `docs/技术难点与解决方案.md` 的难点 4。

必须明确以下映射关系：

- `goal.md` 对应 `spec.md` 的需求目标与边界
- `plan.md` 对应 `design.md` 的技术方案摘要与变更范围
- `steps.md` 对应 `tasks.md` 的执行任务拆解
- `acceptance.md` 对应 `verify.md` 的验证与验收结果

要求：

- 小白版是“对外审核入口”
- AI 执行版是“对内执行入口”
- 不能只保留一套并口头声称兼顾两者
- 不能让小白继续直接以 `design.md / tasks.md` 为主入口
- 不能让 AI 只看口语化文档而失去技术执行精度
- 不要求两套文档在字面表达上逐字一致，但必须建立**一致性检查机制**
- 在 `/verify` 或等价验证阶段，必须检查小白版与 AI 执行版是否出现目标、范围、验收口径上的实质分叉
- 若出现实质分叉，必须先修正文档，再进入后续实现或验收阶段

## 六、上下文 / token 成本控制要求

这套体系必须优先考虑 Claude Code 的上下文窗口成本，避免后续执行时需要读取大量预置内容导致：

- 上下文膨胀
- 成本升高
- 遵守率下降
- 自动执行不稳定

### 必须形成的结构

1. **最小常驻入口**
   - `CLAUDE.md` 只保留最小必需规则
   - 目标控制在约 200 行以内
   - 不要把整份长需求直接塞入常驻上下文

2. **规则分层**
   - 高频、全局规则进入 `CLAUDE.md` 或无 `paths` 的 rules
   - 场景化、目录化规则进入 `.claude/rules/`
   - 低频长说明转为 skills 或说明层文档

3. **按需加载**
   - 只有在处理特定文件、特定目录、特定阶段时，才加载对应规则
   - 不得把所有阶段规则全量常驻

4. **自动预处理**
   - 大日志、大量测试输出、大量变更摘要，不应直接原样进入主上下文
   - 第一阶段只要求完成**过滤、截断、压缩**
   - 第二阶段如确有必要，才允许引入更重型的摘要能力
   - 必须优先设计成由 hooks、脚本或 agent 先做结构化预处理，再交给 Claude

### 设计原则

必须形成：

**最小常驻 + 条件加载 + 工具化复用 + 自动预处理**

## 七、Claude Code 实际可加载、可执行要求

设计结果不能只是“内容正确”，还必须保证 Claude Code 真正能识别和使用。

必须同时设计：

- 文件放置位置
- 文件命名规范
- 哪些内容放在 `CLAUDE.md`
- 哪些内容放在 `.claude/rules/`
- 哪些内容做成 skills
- 哪些内容通过 hooks 触发
- 哪些内容适合 agents / subagents 持有
- 哪些内容进入 settings
- 如何验证这些内容已经被 Claude Code 正确纳入执行体系

要求：

必须避免出现“文档写出来了，但因为路径错误、命名错误、粒度错误、放置位置错误，导致 Claude Code 根本不读取或不触发”的情况。

## 八、命令 / Skills / Agents 的要求

### 1. 技术方向

后续应采用：

- **skills 优先**
- `commands` 兼容保留
- agents / subagents 明确职责
- hooks 提供硬门禁与记录能力

### 2. 命令体系固定规则（强制锁死）

本项目命令体系**只允许扁平命令**，固定为：

- `/spec`
- `/design`
- `/tasks`
- `/implement`
- `/verify`

要求：

- **禁止** `/project:*` 作为本项目实现方案
- 禁止在说明文档、命令文档、技能文档、agent 文档中继续引用 `/project:*`
- 现有遗留文案如出现 `/project:*`，必须在迁移阶段统一替换为扁平命令
- command 文件命名、实际调用名、文档说明三者必须完全一致

### 3. 命令或 Skill 文件元信息要求

后续命令或 skill 文件不得只写裸正文。

至少必须补全：

- `description`
- `argument-hint`
- 必要时 `allowed-tools`
- 必要时 `model`
- 必要时触发方式说明

### 4. 高风险 skill 触发控制（新增硬约束）

以下高风险 skill 默认**不得允许 Claude 自动调用**，必须手动触发：

- `implement`
- `verify`
- `fix-loop`
- `deploy`
- 任何具有写入、批量修改、执行验证闭环、副作用提交能力的 skill

要求：

- 这些高风险 skill 必须设置 `disable-model-invocation: true`
- 必须由用户手动触发
- 不允许 Claude 自行判断时机并自动启动
- 低风险 skill 是否允许自动调用，必须在设计时单独标明

### 5. 背景知识类与目录专用 skill 约束（新增硬约束）

必须明确：

- 背景知识类、内部参考类、只用于被其他流程读取的 skill，必须设置 `user-invocable: false`
- 目录专用、阶段专用、功能专用 skill，必须配置 `paths`
- 不允许把仅对局部目录有效的 skill 设计成全局可见、全局可触发
- 如 skill 属于重型研究、长上下文、分叉式工作流，必须显式说明是否使用 `context: fork` 或等价上下文隔离策略，以及是否绑定特定 agent

### 6. command 与 skill 冲突规则（新增硬约束）

必须明确：

- **禁止同名 skill 与 command 并存**
- 若保留 command，只能作为兼容别名层
- command 不得与目标 skill 形成歧义冲突
- 必须提供 command → skill 的显式映射表
- 后续优先维护 skill，兼容 command 仅用于迁移期保留

### 7. Windows shell 策略（新增硬约束）

必须明确：

- 默认不允许在 skill 中使用内联 `!` shell 作为主实现路径
- 默认优先使用 Node.js 脚本
- 若确实必须使用内联 shell：
  - 必须显式声明 `shell: powershell`
  - 必须说明对应环境前置条件已满足
- 不允许生成 bash-only 的 skill、hook、检查命令或关键执行路径
- Windows 是第一优先测试平台，所有关键执行能力必须先满足 Windows 可运行

### 8. Agents / Subagents 要求

后续若保留或增强 agent 体系，必须明确：

- 名称
- 描述
- 什么时候必须主动使用
- 什么时候允许自动委派
- 工具范围
- 输出产物
- 最大工作边界
- 与小白版 / AI 执行版的关系

对于以下强职责 agent，描述中必须明确“主动使用”或“必须在特定阶段使用”：

- spec-writer / 需求分析类
- designer / 架构设计类
- reviewer / 验收审查类
- tester / 测试验证类
- implementer / 实现执行类
- planner / 计划编排类

## 九、必须新增机器可读状态文件

如果想支持自动闭环，仅靠 Markdown 不够稳定。

每个功能目录必须有一个机器可读状态文件：

- **`feature.json`**

要求：

- 本项目状态文件**只允许 `feature.json`**
- **禁止** `feature.yaml`
- hooks、rules、agents、skills、commands 都只认 `feature.json`
- 同一项目内不得出现 YAML / JSON 混用状态文件

至少应包含：

- `feature_id`
- `status`：`draft / review / approved / implementing / verifying / done / blocked`
- `allowed_paths`
- `forbidden_paths`
- `required_checks`
- `max_fix_rounds`
- `repeat_failure_stop_threshold`
- `human_review_required_when`
- `stop_conditions`
- `evidence_dir`
- `owner`
- `updated_at`
- `phase`
- `phase_gate_approved`
- `phase_gate_approved_by`
- `phase_gate_approved_at`
- `phase_gate_note`

### 1. feature-id 命名规范（新增硬约束）

必须明确：

- `feature_id` 必须使用 `kebab-case`
- 禁止中文目录名作为执行态功能目录名
- 所有相对路径默认相对于项目根目录解析
- 执行态目录路径统一为：`features/<feature-id>/`

要求：

Claude 的自动实现、自动验证、自动修复，不得只靠自由发挥推断，必须读取这份机器可读状态文件。

### 2. required_checks 结构化 schema（新增硬约束）

`required_checks` 不得再使用模糊描述，必须采用结构化 schema。

每一条检查至少应包含：

- `id`
- `name`
- `shell`
- `command_windows`
- `command_bash`（可选兼容字段）
- `cwd`
- `timeout_sec`
- `allow_network`
- `evidence_file`
- `stop_on_fail`

要求：

- 自动验证循环只能运行在 `required_checks` 中声明过的检查项
- 不允许靠自由文本猜测要执行什么命令
- hooks 必须基于结构化字段判断，而不是字符串模糊匹配
- `command_windows` 是权威执行字段
- `command_bash` 仅用于兼容非 Windows 环境，允许为空
- Windows 平台必须优先按 `command_windows` 执行与校验
- 若同时提供 `command_windows` 与 `command_bash`，两者语义必须等价

### 3. 状态机完整约束（新增硬约束）

本项目状态机必须固定为：

- `draft -> review -> approved -> implementing -> verifying -> done | blocked`

要求：

- 不允许跳过中间状态
- 不允许从 `draft` 直接进入 `approved`
- 不允许从 `approved` 直接进入 `done`
- 非法状态跳转必须被拒绝，并记录原因

### 4. 状态推进职责

必须明确谁可以推进哪些状态：

- `draft -> review`
  - 可由 Claude 在文档齐备后推进
  - 但必须写回 `feature.json`

- `review -> approved`
  - 只能由人工批准动作推进
  - Claude 不得自行推进到 `approved`

- `approved -> implementing`
  - 可由 Claude 在满足全部实现前置条件后推进
  - 必须记录启动实现时间与依据

- `implementing -> verifying`
  - 可由 Claude 在实现动作完成且进入验证阶段时推进
  - 必须记录本轮改动摘要

- `verifying -> done`
  - 只能在全部必需检查与声明的 CI 检查通过后推进
  - 必须记录验证结论

- `verifying -> blocked`
  - 当达到停止条件、人工接管条件或失败升级条件时推进
  - 必须记录阻塞原因

## 十、批准状态的进入条件

> ⚠️ **技术难点参考**：本节的 14 项前置检查涉及 hook 脚本路径反推和状态机校验，实现时请参考 `docs/技术难点与解决方案.md` 的难点 1 和难点 3。

“spec 审核通过后”必须有可执行判定，不允许模糊表述。

只有同时满足以下条件，才允许进入实现阶段：

1. `goal.md` 已存在
2. `plan.md` 已存在
3. `steps.md` 已存在
4. `acceptance.md` 已存在
5. `spec.md` 已存在
6. `design.md` 已存在
7. `tasks.md` 已存在
8. 机器可读状态文件存在
9. 状态文件中的 `status = approved`
10. 本次允许修改路径已声明
11. 本次禁止修改路径已声明
12. 验证命令已声明
13. 停止条件已声明
14. 人工接管条件已声明

如果任一条件缺失，必须阻止开始实现。

### 1. 批准协议（新增硬约束）

`approved` 不得是模糊状态，必须有明确批准协议。

状态文件中必须增加：

- `approved_by`
- `approved_at`
- `approval_note`
- `approval_method`

## 十二、Hooks 职责边界（必须明确）

> ⚠️ **技术难点参考**：本节所有 hook 的实现涉及 Windows 兼容性、路径反推、阻断计数持久化等难点，实现时请参考 `docs/技术难点与解决方案.md` 的难点 1、2、5、6。特别注意：hook 的 `if` 条件必须覆盖 `MultiEdit`（见难点 6 补充硬约束）。

后续交付不能只写”建议用 hooks”，而必须明确每类 hook 的用途和阻断级别。

### 0. hooks 注册要求

`.claude/hooks/` 可以作为 hook 脚本存放目录，但**仅有目录存在不代表已生效**。  
所有 hooks 必须在 `.claude/settings.json` 中显式注册后，才视为真正进入执行体系。

对写入类工具的 hook 覆盖范围必须至少包含：

- `Edit`
- `MultiEdit`
- `Write`

不得遗漏 `MultiEdit`，否则批量修改会绕过写入门禁与证据记录。

至少覆盖：

### 1. `UserPromptSubmit`
职责：

- 检查用户请求是否超出当前阶段
- 必要时阻断无效请求
- 必要时补充上下文摘要
- 必要时提醒当前 feature 状态不允许执行

类型区分：

- 可有提示型
- 可有阻断型

### 2. `PreToolUse(Edit|MultiEdit|Write)`
职责：

- 检查 feature 是否已批准
- 检查是否缺少小白四件套或 AI 执行版核心文档
- 检查是否缺少状态文件
- 检查是否越出允许修改范围
- 不符合条件时阻断写入
- hook 条件必须同时覆盖 `Edit`、`MultiEdit`、`Write`

类型区分：

- 必须有阻断型

### 3. `PreToolUse(Bash)`
职责：

- 检查命令是否属于允许执行的验证命令白名单
- 检查是否存在当前阶段允许的自动修复命令
- 缺少白名单时阻断

类型区分：

- 必须有阻断型

### 4. `PostToolUse(Edit|MultiEdit|Write)`
职责：

- 记录本轮改动文件清单
- 输出到证据目录
- 更新迭代日志
- hook 条件必须同时覆盖 `Edit`、`MultiEdit`、`Write`

类型区分：

- 记录型，不负责回滚

### 5. `PostToolUse(Bash)`
职责：

- 收集 lint / typecheck / test / smoke 结果摘要
- 写入检查记录
- 供自动修复循环读取

类型区分：

- 记录型

### 6. `Stop` / `SubagentStop`
职责：

- 检查是否满足停止条件
- 不满足则阻止停止并要求继续
- 达到上限则升级为人工接管

类型区分：

- 必须有阻断型

### 7. Stop hook 逃生口（新增硬约束）

为避免 hook 自身配置错误导致会话无法退出，必须增加紧急放行机制。

状态文件中必须支持：

- `override_stop_guard`
- `max_stop_blocks`

要求：

- 当 `override_stop_guard = true` 时，允许停止保护被临时绕过
- 连续阻断超过 `max_stop_blocks` 后，必须自动降级为人工接管
- 停止保护不得无限循环阻断
- 必须记录本次逃生口触发原因与时间

## 十三、自动闭环的边界保护

> ⚠️ **技术难点参考**：Bash 双层策略和 Stop 逃生口的实现，请参考 `docs/技术难点与解决方案.md` 的难点 2 和难点 5。

自动计划、自动开发、自动验证、自动修复、自动迭代可以做，但必须有边界。

至少必须包含：

1. `max_fix_rounds`
2. 同一错误重复出现 N 次后停止
3. 新增失败数上升时停止
4. 越界修改迹象出现时停止
5. 缺少可执行验证命令时停止
6. 涉及以下情况时请求人工确认：
   - 数据库迁移
   - 删除数据
   - 批量改路径
   - 权限提升
   - 外部发布
   - 高风险脚本执行

### 1. 自动修复轮次约束

必须明确：

- 默认 `max_fix_rounds = 3`
- 项目级允许调整，但不得超过 `5`
- 超过上限必须强制人工接管

### 2. Bash 执行控制策略

不得简单采用“所有 Bash 全白名单”或“完全自由执行”。

必须采用双层策略：

- **自动验证循环中的命令**：只能来自 `required_checks` 的结构化白名单
- **普通探索性命令**：允许有限探索，但必须有高危命令黑名单与风险拦截
- 对涉及删除、发布、权限提升、批量改动的命令，必须人工确认

在越过允许改动范围时，写文件行为会被阻断
在缺少验证命令时，自动修复循环不会启动
在达到最大修复轮次时，会停止并升级为人工接管
小白版与 AI 执行版内容能互相映射，不出现长期分叉
高风险 skill 无法被 Claude 自动触发
specs/ 不再被当作执行态入口使用
项目中不存在未经批准的嵌套规则源
feature.json 是唯一状态文件格式
写入门禁对 `Edit`、`MultiEdit`、`Write` 都生效
`command_windows` 被当作权威字段执行，`command_bash` 仅作为兼容字段
当前 feature 运行在 `feature/<feature-id>` 工作分支
进入 `done` 前，声明的 CI 检查已经通过并留有证据
第一阶段未获得阶段确认前，不会自动进入第二阶段

这些验证步骤必须成为正式交付的一部分。
