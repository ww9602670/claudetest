- 任务拆解（按优先级）

## 六、真实工程入口定义

### 6.1 真实工程入口的定义原则

**什么是真实工程入口**：
- 能够真实执行的最小工程动作
- 能够验证代码基本可用性的命令
- 不依赖尚未建立的脚本或工具
- 可在本地和 CI 中重复执行

**不是真实工程入口**：
- 假设存在但尚未创建的测试脚本
- 需要复杂环境配置的命令
- 无法自动化执行的手动操作

### 6.2 required_checks 结构约束

每个 check 必须包含以下字段：

```json
{
  "id": "唯一标识",
  "name": "检查名称",
  "shell": "cmd 或 bash",
  "command_windows": "Windows 命令",
  "command_bash": "Linux/Mac 命令",
  "cwd": "执行目录（相对于仓库根目录）",
  "timeout_sec": 超时秒数,
  "allow_network": false,
  "evidence_file": "证据文件路径（相对于 feature 目录）",
  "stop_on_fail": true/false
}
```

### 6.3 首轮实施中如何建立 required_checks

**阶段 1：planning 阶段**

在 spec 三件套完成后，根据 design.md 中的技术选型，定义最小可执行检查：

- 如果是 HTML/CSS/JS：可能包括文件存在性检查、语法检查
- 如果是其他技术栈：根据实际情况定义

**阶段 2：implementing 阶段**

随着代码实施，逐步建立真实可执行的检查：

- 先建立最基础的检查（如文件读取、语法验证）
- 再建立功能性检查（如启动验证、基本交互）
- 如需测试脚本，先创建脚本，再添加到 required_checks

**重要原则**：
- 不要在 required_checks 中引用尚不存在的脚本
- 如果需要某个脚本，先创建它，再添加检查
- 如果某个检查暂时无法建立，记录原因到 evidence

### 6.4 最小工程动作的候选类型

根据技术选型，以下是可能的最小检查类型：

**对于 HTML/CSS/JS 项目**：
- 文件存在性检查：`test -f src/index.html` 或 `type src\index.html`
- JS 语法检查：`node --check src/game.js`（如果有 Node.js）
- HTML 基本验证：读取文件内容确认非空

**对于其他技术栈**：
- 根据实际情况定义
- 优先使用语言自带的检查工具
- 避免依赖复杂的第三方工具

### 6.5 当预设命令/脚本不存在时的处理方式

1. **评估是否必需**：该检查是否对验证机制必要
2. **尝试建立**：如果必需且可行，创建相应的脚本或命令
3. **降级替代**：如果无法建立，使用更简单的替代检查
4. **记录原因**：在 evidence 中记录为何无法建立及替代方案
5. **不强行推进**：如果核心检查无法建立且无替代方案，记录 blocked

## 七、evidence 要求

### 7.1 必需的 evidence 类型

1. **命令执行证据**：每个 required_checks 的执行输出
2. **状态变更证据**：feature.json 状态变更记录
3. **CI 执行证据**：CI 运行结果与链接
4. **复审证据**：阶段性复审结论
5. **调整记录**：当实际实施偏离任务包预设时的调整说明

### 7.2 evidence 文件命名原则

- 使用描述性名称，能够反映证据内容
- 包含时间戳（如适用）
- 与 required_checks 中的 evidence_file 字段对应
- 遵循一致的命名规范

**示例**：
```
evidence/
├── check-[检查类型]-[时间戳].txt
├── ci-run-[时间戳].md
├── status-change-[时间戳].md
├── review-[时间戳].md
└── adjustment-[调整内容]-[时间戳].md
```

### 7.3 evidence 与 required_checks 映射

每个 required_checks 必须有对应的 evidence 文件，映射关系必须可追溯。

在 required_checks 的 evidence_file 字段中明确指定证据文件路径。

### 7.4 调整记录的 evidence 要求

当实际实施与任务包预设不一致时，必须记录：

- 预设内容是什么
- 实际情况是什么
- 为何需要调整
- 调整后的方案
- 调整是否在允许范围内

## 八、自动修复边界

### 8.1 允许自动修复的范围

Sonnet 可以自动修复以下问题：

1. **语法错误**：代码语法错误
2. **简单逻辑错误**：变量名错误、条件判断错误
3. **样式问题**：界面样式调整
4. **文档错误**：拼写错误、格式问题
5. **路径问题**：文件路径引用错误

### 8.2 禁止自动修复的范围

以下情况必须停止并记录 blocked：

1. **需求不明确**：spec 中需求描述不清晰
2. **设计冲突**：实施与设计文档冲突
3. **环境问题**：运行环境缺失或配置问题
4. **超出范围**：需要后端服务、数据库等超出边界的能力

### 8.3 修复次数限制

- 同一问题最多尝试修复 3 次
- 3 次后仍未解决，必须记录 blocked 并停止
- blocked 记录必须包含：问题描述、已尝试方案、建议解决方向

## 九、Sonnet 可执行范围

### 9.1 Sonnet 可以独立完成的任务

1. **spec 三件套编写**：基于目标编写 requirements.md、design.md、tasks.md
2. **代码实施**：根据 spec 编写代码
3. **required_checks 定义与执行**：定义检查项并运行检查命令
4. **evidence 收集**：将执行结果归档到 evidence 目录
5. **状态推进**：更新 feature.json 状态
6. **简单修复**：修复语法错误、简单逻辑问题
7. **文档产出**：编写非程序员可读说明
8. **调整记录**：记录实施过程中的合理调整

### 9.2 Sonnet 需要人工确认的任务

1. **需求变更**：spec 需要重大调整时
2. **blocked 解除**：遇到阻塞需要人工决策
3. **done 判定**：最终完成需要人工复审

### 9.3 Sonnet 禁止执行的任务

1. **跳过检查**：不得跳过 required_checks
2. **伪造证据**：不得编造 evidence
3. **强制推进**：检查失败时不得强制标记 done
4. **超出边界**：不得扩展到其他业务或 05 能力
5. **突破核心约束**：不得违反第二章 2.4 中的"不可突破的边界"

## 十一、执行流程

### 11.1 阶段划分

**阶段 1：规划（planning）**
1. 创建 feature 目录结构
2. 编写 spec 三件套
3. 根据 design.md 定义初步 required_checks
4. 编写初始非程序员说明
5. 状态：planning → implementing

**阶段 2：实施（implementing）**
1. 根据 tasks.md 逐个实施任务
2. 编写代码
3. 建立真实可执行的 required_checks
4. 本地执行 required_checks
5. 收集 evidence
6. 状态：implementing → verifying

**阶段 3：验证（verifying）**
1. 执行所有 required_checks
2. 触发 CI 检查
3. 验证 evidence 完整性
4. 完善非程序员说明
5. 验证首轮机制验证完成标准
6. 状态：verifying → done 或 blocked

### 11.2 每个阶段的交付物

| 阶段 | 必需交付物 | 验收标准 |
|------|-----------|---------|
| planning | spec 三件套、feature.json、初始说明 | spec 完整一致、结构清晰 |
| implementing | 代码文件、本地检查证据 | 代码可运行、基本功能实现 |
| verifying | 完整 evidence、CI 记录、最终说明 | 所有检查通过、文档完整 |

### 11.3 执行过程中的收敛原则

- 从 spec 出发，不预设具体实现
- 根据实际情况收敛技术方案
- 优先选择简单可行的方案
- 记录重要的收敛决策到 evidence

### 12.2 提交规范

- `style`: 代码格式调整
- `refactor`: 重构

### 12.3 提交要求

- 每个提交必须有清晰的描述
- 提交必须是原子性的（一个提交只做一件事）
- 提交前应通过本地 required_checks（如已建立）

## 十三、CI 最小闭环

### 13.1 CI 配置原则
