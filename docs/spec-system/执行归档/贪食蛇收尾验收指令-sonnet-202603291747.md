你现在进入“收尾补验证”阶段。
注意：本次不是继续开发新功能，不是重建 spec，不是重做架构，而是在**既有成果基础上做最终补验证与收尾确认**。
必须沿用当前项目内已经存在的代码、页面、spec 骨架、文档与目录结构开展工作，不得另起炉灶。

# 一、任务目标

对当前项目已完成内容做一次**补充验证 + 收尾确认**，输出一份可供人工验收的最终验证结果，重点回答以下问题：

1. 当前成果是否真的“能运行、能操作、能通过基础验收”
2. 是否只是“页面能打开”，还是“核心交互链路真实可用”
3. 是否仍存在未验证、假验证、仅代码完成但未实际确认的部分
4. 是否还有阻塞上线/交付的问题
5. 现在是否可以进入“确认完成”状态，若不可以，差什么

# 二、执行边界

你必须遵守以下边界：

1. **不得新建一套 spec**
2. **不得重写已有实现**
3. **不得擅自扩大任务范围**
4. 允许做的事情只有：

   * 阅读现有代码与文档
   * 启动现有项目
   * 进行页面/功能补验证
   * 修正极小范围、明显阻碍验证的问题（仅限必要的小修）
   * 输出验证报告
5. 如果发现问题较大，**只记录，不大改**
6. 本次重点是“验证”和“收尾”，不是“继续开发”

# 三、补验证重点

请按下面顺序执行，不要跳步。

## 1）先确认项目当前可运行状态

需要确认：

* 项目是否能正常启动
* 启动命令是什么
* 启动后访问地址是什么
* 页面是否能正常打开
* 是否存在明显报错、白屏、资源缺失、构建失败
* 当前运行的是开发环境、静态页面，还是完整可交互页面

如果项目无法启动：

* 先排查是否是依赖、脚本、路径、端口、配置问题
* 只允许做最小修复
* 修不动就停止扩改，直接记录原因

## 2）补验证“不是只能打开，而是真的能操作”

必须实际检查以下内容：

* 页面是否可见
* 核心按钮/输入/交互是否真的可操作
* 点击后是否有真实反馈
* 是否只是静态展示，没有真正逻辑
* 是否存在“看起来能点，实际上没效果”
* 是否存在“页面局部能用，但核心流程断掉”

如果是类似 snake-game 这种页面，则至少验证：

* 页面成功加载
* 游戏能开始
* 键盘/按钮操作是否真实生效
* 游戏状态是否会变化
* 基本交互是否闭环
* 是否存在明显异常（无法操作、刷新即坏、卡死、逻辑错乱）

## 3）检查是否存在“伪完成”

重点排查：

* 代码写了，但没接入
* 页面做了，但入口未连接
* 能运行，但主流程没通
* 只有演示效果，没有实际逻辑
* 只有局部功能可用，验收链路未闭合
* 文档标记完成，但没有对应验证证据

## 4）检查是否存在明显遗留问题

例如但不限于：

* 启动方式不清晰
* 依赖缺失
* 脚本命令错误
* 页面路径不对
* 控制台明显报错
* 交互异常
* 结果与预期不符
* 用户一看就会卡住的地方

## 5）给出“人工还需要看什么”的明确清单

请把需要人工最终确认的内容单独列出来，区分为：

### A. 你已经替人工验证完成的

### B. 必须人工亲自再点一遍确认的

### C. 现在还无法验证的

# 四、输出要求

执行完成后，不要只说“已验证”或者“基本没问题”。
你必须输出一份**结构化收尾补验证报告**，使用下面格式：

---

# 收尾补验证报告

## 1. 本次验证范围

* 本次验证了哪些页面 / 功能 / 文件 / 启动链路

## 2. 运行验证结果

* 是否成功启动
* 启动命令
* 访问地址
* 是否报错
* 是否做了最小修复（如有，写明）

## 3. 核心操作验证结果

按“操作项 -> 结果 -> 说明”列出：

* 哪些操作真实可用
* 哪些只是表面可见
* 哪些失败
* 哪些未验证

## 4. 是否存在伪完成

明确回答：

* 有 / 无
* 如有，具体是什么

## 5. 当前是否达到“可人工验收”状态

只允许三种结论之一：

* 可以直接人工验收
* 可以人工验收，但有残留风险
* 暂不能验收

并说明理由。

## 6. 人工最终只需要检查什么

只列最关键、最少量、非程序员也能看懂的检查项。

## 7. 风险与遗留问题

* 阻塞型
* 非阻塞型

## 8. 最终结论

用最直白的话说明：

* 这个项目现在到底算没算真正完成到可交付程度
* 如果不算，最差的那一口气卡在哪

---

# 五、执行原则

1. 不要泛泛而谈
2. 不要只看代码不运行
3. 不要只开页面不操作
4. 不要因为看到页面能动，就默认全部完成
5. 必须把“已验证”“未验证”“验证失败”分开写清楚
6. 所有结论都要基于实际检查结果，不允许想当然

# 六、如果你发现能顺手补齐的小问题

仅当同时满足以下条件时，才允许顺手修：

* 修改范围非常小
* 不会引入结构性变化
* 目的是为了完成验证
* 修完能立刻重新验证

修完后必须说明：

* 改了什么
* 为什么改
* 改完验证结果如何

# 七、特别要求

请把结果写得让非程序员也能看懂。
尤其是“当前到底能不能算完成”“人工还要再看什么”，必须写得非常直接，不要模糊。
