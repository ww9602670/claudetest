# 文档生命周期与归档规则

> 版本：v1.0  
> 创建日期：2026-03-28  
> 用途：规定项目文档的分类、保留策略、失效判定与 Claude 执行优先级，避免旧材料污染新执行。

---

## 一、为什么需要这份规则

项目开发过程中产出了大量文档，类型包括：执行指令、阶段需求、阶段审查报告、修复报告、复验报告、收口报告、验收清单、临时诊断材料等。

如果不做分层，后续模型执行时会面临以下风险：
- 把已失效的审查报告当成当前执行基线
- 把临时诊断材料当成长期设计依据
- 把旧版阶段报告覆盖新版结论

本规则的目标是：**让每份文档的"是否有效"一眼可辨，让 Claude 每次执行前知道该读哪些、不该读哪些。**

---

## 二、文档分类体系

### 第一类：当前权威基线文档（始终有效）

这些文档是当前执行的核心依据，任何时候都应优先读取，不得被其他类型文档替代或覆盖。

| 文档 | 说明 |
|------|------|
| `docs/需求.md` | 项目总需求，长期有效 |
| `docs/spec-system/阶段需求/当前激活的阶段文档` | 当前正在执行的那一份阶段需求 |
| `docs/spec-system/验收清单.md` | 当前签收版验收清单 |
| `CLAUDE.md` | Claude 执行基线规则 |

**保留策略：必须长期保留，不得删除，不得降级。**

---

### 第二类：阶段归档文档（阶段完成后归档）

这些文档记录了某个阶段的执行结果和判定依据，阶段完成后仍保留，但不再作为下一阶段的执行输入。

| 典型文件 | 说明 |
|----------|------|
| `docs/spec-system/阶段需求/01-*.md`（已完成阶段） | 已完成阶段的需求文档 |
| `docs/收口执行报告.md` | 阶段收口产出，记录实际落地内容 |
| `docs/spec-system/阶段需求/01-阶段审查报告.md` | 特定阶段的审查结论 |
| `docs/spec-system/验收清单审查报告.md` | 验收过程的阶段性判定 |

**保留策略：可归档保留（审计用），但不得再作为当前执行依据。**

---

### 第三类：临时执行文档（用后可清理）

这些文档是某次特定任务的执行指令或辅助材料，任务完成后即可清理。

| 典型文件 | 说明 |
|----------|------|
| `docs/执行指令-opus.md` | 特定模型的一次性执行指令 |
| `docs/执行指令-sonnet.md` | 特定模型的一次性执行指令 |
| `docs/执行指令.md` | 通用临时执行指令 |
| `docs/继续执行指令.md` | 某次中断恢复用的临时指令 |
| `docs/收口执行指令.md` | 某次收口操作的临时指令 |
| `docs/审核指令-opus.md` | 某次审核操作的临时指令 |
| `docs/修复指令-opus.md` | 某次修复操作的临时指令 |

**保留策略：任务完成后可删除。如需存档，移入 `docs/spec-system/执行归档/` 目录。**

---

### 第四类：废弃 / 历史参考文档（仅审计价值）

这些文档已被新版本内容吸收或覆盖，不再具备执行价值，仅保留审计追溯用。

| 典型文件 | 说明 |
|----------|------|
| `docs/修复报告.md` | 修复过程记录，已被最终收口报告吸收 |
| `docs/gate-write-修复报告-迭代记录.md` | 迭代过程记录，已完成 |
| `docs/hooks不生效-终极诊断报告.md` | 历史诊断，已解决 |
| `docs/gate-write-故障诊断与修复方案.md` | 历史诊断，已解决 |
| `docs/第一阶段审计报告.md` | 历史阶段产出 |
| `docs/chanchu.md` | 临时诊断材料 |

**保留策略：不得再作为当前执行依据。可选择移入 `docs/spec-system/审查归档/` 或直接删除。**

## 三、Claude 执行优先级规则

### 3.1 每次执行前，Claude 默认应读取的文档

按以下顺序读取，不得跳过前两项：

1. **`CLAUDE.md`** — 执行基线规则（第一优先）
2. **当前激活的阶段需求文档** — 从 `docs/spec-system/阶段需求索引.md` 或 `docs/spec-system/需求实现现状与阶段拆分报告.md` 确认当前激活阶段
3. **`docs/spec-system/验收清单.md`** — 当前签收版验收清单

### 3.2 以下文档即使存在，也不应作为当前执行基线

| 不应作为基线的文档 | 原因 |
|--------------------|------|
| 已完成阶段的需求文档 | 该阶段已结束，继续读取会导致重复执行 |
| 旧版审查报告 | 可能被新版报告覆盖，结论已过期 |
| 临时执行指令文档 | 仅限单次任务使用，不代表长期策略 |
| 临时诊断材料 | 问题已解决，不再反映当前状态 |
| 废弃类文档 | 结论已被吸收，继续读取会引入矛盾 |

### 3.3 如何避免"旧审查报告覆盖新结论"

- 审查报告必须在文件名或文件头注明**所属阶段编号**和**生效日期**
- 如有多份审查报告，以**文件名中编号最大 / 日期最新**的为准
- 收口报告一旦生成，对应的过程性审查报告自动降级为"阶段归档"

## 四、目录建议

建议逐步将文档按以下目录结构存放（不要求立刻迁移，新产出文件开始遵守即可）：

```
docs/
├── 需求.md                          ← 第一类：权威基线，长期不动
├── spec-system/
│   ├── 文档生命周期与归档规则.md     ← 本文件（权威基线）
│   ├── 需求实现现状与阶段拆分报告.md ← 第一类：权威基线
│   ├── 验收清单.md                   ← 第一类：权威基线
│   ├── 阶段需求/                     ← 第一类（当前激活阶段）/ 第二类（已完成阶段）
│   │   ├── 01-现状收敛与口径统一.md  ← 已完成，归档
│   │   ├── 02-第一阶段余量收尾.md    ← 当前激活（示例）
│   │   └── ...
│   ├── 审查归档/                     ← 第二类 / 第四类：阶段报告存放处
│   │   └── 01-阶段审查报告.md
│   ├── 执行归档/                     ← 第三类归档：完成后的执行指令
│   │   └── 执行指令-20260328.md
│   └── 临时材料/                     ← 第三类临时：当前在用的临时指令
│       └── 执行指令-sonnet.md
```

> **注意：不要求立刻移动所有文件。只需：新产出文档按此规则存放，旧文档逐步归档。**

## 五、文件命名建议

命名规则需让非程序员一眼看出文档状态：

| 文档状态 | 命名建议 | 示例 |
|----------|----------|------|
| 当前有效 | 简洁名称，无特殊后缀 | `验收清单.md` |
| 历史归档 | 加阶段编号前缀 | `01-阶段审查报告.md` |
| 临时材料 | 加执行日期或模型标识 | `执行指令-sonnet-20260328.md` |
| 已失效 | 加 `[失效]` 前缀或移入 `审查归档/` | `[失效]修复报告.md` |

## 六、失效判定规则

满足以下任意一条，该文档即可判定为失效：

1. **已被新版阶段审查报告覆盖**  
   例：阶段 02 的审查报告发布后，阶段 01 的过程审查报告自动失效。

2. **已被收口执行报告吸收**  
   例：收口报告明确记录了某个修复结论，对应的修复报告即失效。

3. **不再作为后续阶段的输入**  
   例：阶段 01 需求文档在阶段 01 完成后，不再作为阶段 02 的执行输入。

4. **问题已解决，诊断文档无执行价值**  
   例：hooks 不生效问题已修复，对应的诊断报告仅保留审计价值。

5. **临时执行指令已执行完毕**  
   例：执行指令-sonnet.md 对应任务完成后，该文件可清理或归档。

**失效文档的处理方式：**
- 优先选择：移入对应的归档目录（`审查归档/` 或 `执行归档/`）
- 次选：在文件头标注 `> ⚠️ 本文档已失效，仅供审计追溯，不得作为执行依据。`
- 最后选择：直接删除（需人工确认后操作）

## 七、与阶段文档的关系

| 文档类型 | 定位 | 说明 |
|----------|------|------|
| `阶段需求/*.md`（当前激活） | 当前执行主入口 | Claude 每次任务开始必须读取，是最重要的执行依据 |
| `阶段需求/*.md`（已完成阶段） | 阶段归档 | 仅供追溯，不得再作为执行输入 |
| 阶段审查报告 | 阶段完成判定依据 | 用于判断某阶段是否真正完成，完成后降级为归档 |
| 收口执行报告 | 阶段实际产出记录 | 比需求文档更准确地反映实际落地内容 |
| 临时执行指令 | 单次任务输入 | 不应长期混入主执行层，任务完成后清理或归档 |
| 验收清单 | 阶段完成判定的结构化依据 | 属于权威基线，长期有效，随阶段推进持续更新 |

---

## 八、规则执行的最小建议

**现在不需要做什么：**
- 不需要立刻整理所有历史文档
- 不需要自动化归档脚本
- 不需要一次性迁移所有文件

**从现在开始需要做什么：**
1. 新产出的文档，按本规则存放到正确目录
2. 每次给 Claude 下发任务前，明确告知"当前激活阶段是哪个"
3. 一个阶段完成后，人工将该阶段的审查/收口报告标注或移入归档目录
4. Claude 执行时，如发现正在读取的文档与本规则冲突，应优先遵循本规则
