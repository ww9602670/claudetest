## 二、任务清单

## 04-T1 启动条件与前置口径修订

- 任务目标：消除“启动许可条件”与“blocked 处理动作”混写冲突
- 修改对象：`04-第二阶段工程化接入.md`
- 修改范围：启动条件、阻塞处理、前置分层章节
- 输入：04 审查报告、01~03 审查结论
- 输出：启动许可、禁止启动、预启动 blocked、执行中 blocked 的分离定义
- 依赖关系：无（首任务）
- 验证方式：复审清单中“启动口径冲突已消除”项通过
- 完成标准：复审者可仅根据文档判定“允许启动/不允许启动/仅记录阻塞”
- 失败/阻塞判定：若仍存在语义歧义，任务不通过并阻塞后续任务

## 04-T2 总需求条款映射补齐

- 任务目标：建立 04 与总需求关键条款的一一映射
- 修改对象：`04-第二阶段工程化接入.md`
- 修改范围：定位、范围、边界、完成判定章节
- 输入：`docs/需求.md` 相关条款
- 输出：04 关键能力与总需求条款映射关系
- 依赖关系：依赖 04-T1
- 验证方式：复审清单“总需求映射存在且可追溯”通过
- 完成标准：04 范围无偷换、无无声明缩减
- 失败/阻塞判定：映射缺失或无法对应关键能力时阻塞

## 04-T3 required_checks schema 对齐

- 任务目标：将 `required_checks` 升级为总需求要求的结构化字段
- 修改对象：04 文档中的 schema 章节、相关模板说明
- 修改范围：字段定义、约束、执行口径
- 输入：`docs/需求.md` schema 强约束
- 输出：含 `id/name/shell/command_windows/command_bash/cwd/timeout_sec/allow_network/evidence_file/stop_on_fail` 的最小 schema 定义
- 依赖关系：依赖 04-T2
- 验证方式：复审清单“schema 对齐”通过
- 完成标准：无关键字段缺失，且 `command_windows` 权威地位明确
- 失败/阻塞判定：字段缺失或弱化即不通过
