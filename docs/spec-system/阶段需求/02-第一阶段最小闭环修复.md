# 阶段需求 02：第一阶段最小闭环修复

## 阶段目标

在不扩展第二阶段能力的前提下，把当前第一阶段骨架修到“可重新验收”的状态。

本阶段只处理**阻断签收**的问题，不做审计增强，不做工程化接入。

## 前置条件

必须先完成：

- `docs/spec-system/阶段需求/01-现状收敛与口径统一.md`

如果口径还未统一，本阶段不得开始。

## 本阶段输入

只读取以下文件：

- `docs/spec-system/验收清单.md`
- `docs/spec-system/验收清单审查报告.md`
- `.claude/hooks/gate-write.js`
- `.claude/hooks/gate-bash.js`
- `.claude/rules/governance-core.md`
- `.claude/rules/implementation-gate.md`
- `.claude/rules/validation-loop.md`
- `.claude/settings.json`
- `docs/spec-system/模板/AI执行版/feature.json`
- `features/example-login/*`
- `tests/verify-hooks.ps1`

## 本阶段必须完成

1. 修复 `gate-bash.js` 的白名单绕过问题
   - `verifying` 模式不得只看命令首个词
   - 必须阻断 `&&`、`||`、`;`、管道、重定向等组合命令绕过

2. 修复 `gate-write.js` 与规则文档不一致
   - 批准字段校验语义必须与规则文档一致
   - 至少要明确 `approved`、`implementing`、`verifying` 三个状态的合法写入条件

3. 收口 hooks 注册方案
   - 明确项目级 `.claude/settings.json` 是否承担正式注册
   - 如果运行时仍需用户级配置，必须写清楚“项目内事实源”和“环境接入条件”的边界

4. 修正 `feature.json` 的最低一致版本
   - 先统一字段命名
   - 不要求一次补齐第二阶段 schema
   - 只补到第一阶段闭环所必需的最低集合

5. 处理 `example-login`
   - 若保留占位性质，必须明确标注，且验收清单不能再按真实闭环计分
   - 若升级为真实示例，则必须提供真实证据链

6. 扩充回归验证
   - `tests/verify-hooks.ps1` 必须增加能覆盖本阶段修复点的用例
   - 不能只保留原来的基础 happy path

## 允许修改范围

只允许修改：

- `.claude/hooks/*.js`
- `.claude/rules/*.md`
- `.claude/settings.json`
- `docs/spec-system/**`
- `features/example-login/**`
- `tests/verify-hooks.ps1`
- `CLAUDE.md`

## 禁止事项

- 禁止新增 `PostToolUse` / `Stop` 类 hook
- 禁止新增 CI 配置
- 禁止新增 Git 分支门禁
- 禁止新增完整 agents 集合
- 禁止新增 deploy / release 能力
- 禁止修改业务代码

## 本阶段产出

至少应产出：

1. 修复后的 `gate-write.js`
2. 修复后的 `gate-bash.js`
3. 与实现一致的规则文档
4. 更新后的验收清单或审查报告
5. 更新后的 hook 回归测试

## 完成判定

以下条件同时满足才算完成：

1. 当前第一阶段阻断项被消除或降级为明确的人类决策项
2. hook 逻辑与规则文档不再互相打架
3. 至少一轮 hook 回归测试通过
4. `example-login` 的定位不再含糊

## 需要人工介入的点

1. 若 hooks 注册最终依赖用户级配置，需人工确认是否接受
2. 若 `example-login` 要从占位示例升级为真实示例，需人工确认投入范围
