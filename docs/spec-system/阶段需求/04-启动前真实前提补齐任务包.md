## 四、前置补齐任务清单 (完整版)

### P0-T1: 选定真实 feature

**任务编号**: P0-T1 | **对应阻塞项**: E1  
**是否允许 Sonnet 执行**: 否 (需人工拍板)  
**任务目标**: 确定一个非 demo_only 的真实 feature 作为 04 工程化接入对象

**执行步骤**:
1. 从项目实际需求中筛选候选 feature
2. 评估 feature 的工程代表性  
3. 确认 feature 具备最小可实施性
4. 记录选定理由并产出书面决策

**完成标准**: 产出决策记录文档 (含 feature_id、选定理由)

---

### P0-T2: 创建真实 feature 目录结构

**任务编号**: P0-T2 | **对应阻塞项**: E1  
**是否允许 Sonnet 执行**: 是 (在 feature_id 确定后)  
**任务目标**: 建立真实 feature 的执行态根目录与 8 份基础文档

**修改/创建对象**: `features/<feature-id>/` 及其下文件

**执行步骤**:
1. 创建 `features/<feature-id>/` 目录
2. 使用模板创建小白版 4 件套 (goal/plan/steps/acceptance)
3. 使用模板创建 AI 执行版 4 件套 (spec/design/tasks/verify)
4. 创建 `feature.json` (先填基础字段, `status: draft`)
5. 创建 `evidence/` 目录

**完成标准**: `features/<feature-id>/` 下存在完整 9 个文件/目录

**必需证据**: `ls` 命令输出

---

### P0-T3: 补齐真实 feature.json 并完成 phase2 真实批准

**任务编号**: P0-T3 | **对应阻塞项**: E2  
**是否允许 Sonnet 执行**: 半自动 (Sonnet 准备字段, 人工确认批准)  
**任务目标**: 建立 phase2 真实批准证据, 满足 04 启动前提

**修改/创建对象**: `features/<feature-id>/feature.json`

**执行步骤**:
1. 补齐 `feature.json` 基础字段 (feature_id/owner/allowed_paths/forbidden_paths 等)
2. 补齐 `required_checks` 结构化 schema (10 字段齐全)
3. **人工确认 phase2 批准决策**
4. 填入 phase2 批准字段:
   - `phase: "phase2"`
   - `phase_gate_approved: true`
   - `phase_gate_approved_by` (非空、可追溯)
   - `phase_gate_approved_at` (合法时间戳)
   - `phase_gate_note` (批准范围与条件)
5. 确认 `demo_only: false` 或不设置该字段

**完成标准**: `feature.json` 通过 phase2 真实性检查 (满足 04 文档中的 7 项条件)

**必需证据**: `feature.json` 内容快照 + 核对记录

---

### P0-T4: 确认最小工程入口

**任务编号**: P0-T4 | **对应阻塞项**: E3  
**是否允许 Sonnet 执行**: 是 (命令探测与字段补齐)  
**任务目标**: 确定至少一条可执行的构建/测试/校验命令

**修改/创建对象**: `features/<feature-id>/feature.json` 的 `required_checks`

**执行步骤**:
1. 评估项目实际可用的命令 (lint/typecheck/unit/smoke 中适配项)
2. 选定至少一条作为最小入口
3. 在 `required_checks` 中补齐该命令的结构化字段:
   - `id`/`name`/`shell`/`command_windows` (权威字段)
   - `command_bash`/`cwd`/`timeout_sec`
   - `allow_network`/`evidence_file`/`stop_on_fail`
4. 确认该命令在当前环境可执行

**完成标准**: `required_checks` 中至少存在一条完整的结构化命令

**必需证据**: 命令存在性证明 + 一次 dry-run 输出

---

### P0-T5: 跑通一次真实命令并留存证据

**任务编号**: P0-T5 | **对应阻塞项**: E3  
**是否允许 Sonnet 执行**: 是 (命令执行与证据保存)  
**任务目标**: 产出第一条真实工程执行证据

**修改/创建对象**: `features/<feature-id>/evidence/`

**执行步骤**:
1. 在项目根目录或指定 `cwd` 执行命令
2. 捕获完整输出 (含错误/失败信息, 不得伪造)
3. 将输出保存到 `evidence/<timestamp>-check-results.txt`
4. 记录执行时间戳、命令内容、退出码

**完成标准**: evidence 目录存在真实输出文件

**必需证据**: `<YYYYMMDD>-<HHMMSS>-check-results.txt`

---

### P1-T6: 建立最小 CI 检查链

**任务编号**: P1-T6 | **对应阻塞项**: E4  
**是否允许 Sonnet 执行**: 半自动 (Sonnet 准备配置, 人工确认触发)  
**任务目标**: 建立 lint/typecheck/unit 任一配置的存在与可运行证据

**修改/创建对象**: `.github/workflows/` 或等价 CI 配置

**执行步骤**:
1. 确定项目使用的 CI 平台
2. 创建最小 CI 配置文件 (如 `.github/workflows/check.yml`)
3. 配置至少包含 P0-T4 确定的命令
4. 触发一次 CI 运行 (手动或自动)
5. 保存 CI 运行记录/截图/日志

**完成标准**: CI 配置文件存在且有运行记录

**必需证据**: CI 配置文件 + 运行记录证据

---

### P1-T7: 产出首份真实映射

**任务编号**: P1-T7 | **对应阻塞项**: E5  
**是否允许 Sonnet 执行**: 是 (映射表整理与创建)  
**任务目标**: 产出第一条完整的 `required_checks.id -> CI -> evidence_file` 真实映射

**修改/创建对象**: `features/<feature-id>/evidence/mapping-table.md`

**执行步骤**:
1. 整理 P0-T5 中的 `required_checks.id`

5. 确保三元组可互相追溯

**完成标准**: 映射表文件存在且内容可追溯

**必需证据**: `mapping-table.md`
