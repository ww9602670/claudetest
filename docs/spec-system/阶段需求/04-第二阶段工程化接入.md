## 一、修订目的与适用范围

本修订版用于解决《04 执行前前置条件审查报告》指出的阻塞问题，重点补齐：

1. 启动许可条件与 blocked 处理语义冲突
2. 缺少任务级执行拆解
3. 真实工程环境要求不具备可判定标准
4. `required_checks` schema 与总需求字段不对齐
5. phase2 启动批准真实性定义不清

7. 验收虽有方向但缺少客观证据模板
8. 04→05 稳定交接条件不清

## 二、04 的定位（与总需求一致）

04 的唯一定位是：**建立第二阶段自动化增强前必须稳定的工程化基线**。

### 2.1 与总需求关键条款映射

| 04 基线能力 | 对应总需求要点（`docs/需求.md`） | 04 验收关注点 |
| --- | --- | --- |
| `required_checks` 结构化 schema | 要求 `required_checks` 采用结构化字段，`command_windows` 为权威字段 | 字段齐全、可驱动 hooks/验证/CI |
| `feature.json` 状态机与 `done/blocked` 闭环 | 固定状态机与非法跳转拒绝；`done` 需绑定检查通过 | 状态推进可审计、不得口头判定 done |
| Windows 优先执行策略 | Windows 第一优先，禁止 bash-only 关键路径 | 关键入口可在 Windows 跑通 |
| Git 治理基线 | `feature/<feature-id>` 分支、提交规范、回滚可审计 | 分支一致性、提交信息、回滚策略有证据 |
| CI 最小闭环与 evidence 绑定 | lint/typecheck/unit/smoke 纳入 CI，done 不得仅靠本地成功 | CI 结果可追溯到 evidence |
| 真实示例归属结论 | demo_only 仅占位，不得作为真实闭环依据 | 在 04 内明确“升级或 blocked” |

04 必须完成的基线能力：

- `required_checks` 结构化 schema（以 `command_windows` 为权威执行字段）
- `feature.json` 状态机与 `done / blocked` 正式闭环
- Windows 优先执行策略
- Git 分支/提交/回滚治理基线
- 最小 CI 闭环与 evidence 绑定
- 真实可运行示例归属结论（升级为真实示例，或正式 blocked）

04 不负责的能力（留给 05）：

- `UserPromptSubmit` 控制点
- `Stop / SubagentStop` 与逃生口
- 完整 fix-loop 边界保护
- command / skill 冲突治理深化
- agents / subagents 完整编排

## 三、04 启动许可条件（与启动失败处理分离）

### 3.1 启动许可条件（全部满足才允许进入 04 实施态）

必须同时满足：

1. 01/02/03 阶段文档与审查结论已存在且结论为“可继续下一阶段”
2. 本文档（04 修订版）与 `04-任务拆解与执行计划.md` 已发布并通过人工确认
3. 已存在可用于工程化接入的真实工程载体（非纯占位示例）
4. 至少可识别真实构建/测试/校验入口（可先是最小入口，但必须真实可运行）
5. 待执行 feature 的 `feature.json` 已满足 phase2 启动真实性要求（见第六章）
6. 复审清单（`04-启动前复审清单.md`）阻塞项全部通过

### 3.2 不允许启动的条件（任一命中即禁止进入 04 实施态）

1. 仅有 `demo_only` 示例，未指定真实工程 feature
2. `feature.json` 仍为 phase1，或 phase2 批准字段不完整/不可追溯
3. 真实工程环境最小证据缺失（见第七章）
4. `required_checks` schema 仍停留在弱字段结构
5. 仅有目标宣告，无任务级拆解与验收模板

### 3.3 启动失败时的处理动作（blocked 的语义）

> **blocked 不是“允许启动实施”的替代状态。**
> blocked 在 04 中分两类，必须区分：

1. **预启动 blocked（Pre-Start Blocked）**
   - 触发条件：未满足 3.1 启动许可条件
   - 允许动作：只允许记录阻塞原因、缺失条件、外部依赖
   - 禁止动作：不得进入实施任务、不得宣称 04 在执行

2. **执行中 blocked（In-Execution Blocked）**
   - 触发条件：04 已合法启动后，在实施中遇到不可继续的真实阻塞
   - 要求：必须记录到 evidence 与状态文件，并终止相关任务继续推进

## 六、phase2 启动真实性要求（去“虚化”）

### 6.1 何为真实批准

满足以下全部条件才算真实批准：

1. `phase = "phase2"`
2. `phase_gate_approved = true`
3. `phase_gate_approved_by` 非空且可追溯到真实审批主体
4. `phase_gate_approved_at` 为合法时间戳
5. `phase_gate_note` 说明批准范围与条件
6. 对应 feature 不是 `demo_only: true`
7. 批准记录能关联到复审清单与证据条目

### 6.2 何为无效批准

以下任一情况均视为无效：

- 仅在会话口头宣布“已批准”
- 字段不完整或时间戳非法
- 仍是 phase1 却声称进入 phase2
- `demo_only` 占位示例作为 phase2 启动依据
- “虚拟批准/示例批准”未被显式排除

## 七、真实工程环境最小成立标准

> 当前若不满足下列标准，只能维持“预启动 blocked”，不得进入实施。

### 7.1 最小工程载体标准

1. 存在真实 feature（非 demo_only）
2. 存在可执行的工程命令入口（至少覆盖 lint/typecheck/unit 中可适配项）
3. 能生成真实输出记录（命令输出、失败信息、结果摘要）

### 7.2 最小证据标准

至少需有：

1. 一次真实命令执行记录（非占位文本）
2. 一条与 `required_checks` 映射的检查证据
3. 一条与 CI 结果可互相引用的 evidence 记录

### 7.3 缺失即禁止实施

- 无真实命令入口
- 只有模板/占位输出
- 无法建立 `required_checks -> 命令 -> CI -> evidence` 映射

## 八、required_checks 结构化 schema（对齐总需求）

04 起必须按以下最小字段执行，不得弱化：

- `id`
- `name`
- `shell`
- `command_windows`
- `command_bash`（兼容字段，可为空）
- `cwd`
- `timeout_sec`
- `allow_network`
- `evidence_file`
- `stop_on_fail`

约束：

1. `command_windows` 为权威执行字段
2. 自动验证循环仅可执行 `required_checks` 中声明项
3. 不允许自由文本猜命令
4. hooks/CI/验证脚本必须基于结构化字段判定

---

## 九、CI 与 evidence 绑定规则

### 9.1 最小 CI 检查项

至少声明并执行：

- `lint`
- `typecheck`
- `unit test`
- 必要 `smoke`（若项目适配）

### 9.2 done 与 CI 关系

`verifying -> done` 必须同时满足：

1. `required_checks` 必需项全部通过
2. 声明的 CI 检查全部通过
3. CI 结果已进入 evidence 链并可追溯

任一失败：不得标记 done；按规则停留 `verifying` 或转 `blocked`。

### 9.3 evidence 绑定要求

每个 CI 检查项必须可追溯到：

- 对应 `required_checks.id`
- 对应 evidence 文件路径
- 对应执行时间与结论

具体模板见：`04-验收证据模板.md`。
