# 阶段需求 04：第二阶段工程化接入

## 阶段目标

把治理体系从“第一阶段可审闭环”提升到“第二阶段工程化可执行”。本阶段不仅补上 Git 和 CI 两条工程事实链，还必须把后续自动化增强依赖的工程基线显式接住：

- `required_checks` 结构化 schema
- Windows 优先执行策略
- `feature.json` 状态推进与 `done / blocked` 闭环
- 真实可运行示例的明确归属

本阶段只承接“工程化接入”缺口，不重复 01~03 已完成的口径收敛、最小 gate 修复、evidence 自动沉淀和第一阶段最低 skill 元信息补齐。

## 前置条件

必须同时满足：

1. 已完成以下阶段文档与审查结论：
- `docs/spec-system/阶段需求/01-现状收敛与口径统一.md`
- `docs/spec-system/阶段需求/01-阶段审查报告.md`
- `docs/spec-system/阶段需求/02-第一阶段余量收尾与执行基线增强.md`
- `docs/spec-system/阶段需求/02-阶段审查报告.md`
- `docs/spec-system/阶段需求/03-第一阶段可审闭环增强.md`
- `docs/spec-system/阶段需求/03-阶段审查报告.md`

2. `feature.json` 中已具备明确的第二阶段 phase gate 批准信息。
3. 第一阶段 rules / hooks / evidence 基线已稳定，不再处于返工状态。
4. 当前仓库或目标仓库已具备真实可运行工程环境，至少能给出真实的构建、测试、校验入口。

若不具备真实工程环境，本阶段应直接推进到 `blocked`，并记录缺失前置条件、阻塞原因和所需外部条件；不得伪造 CI 闭环、不得用占位记录冒充工程化完成。

只读取以下内容：

- `docs/需求.md` 中与 `required_checks`、状态机、Windows shell、Git、CI、正式验证清单相关章节
- 第一阶段验收结论：
  - `docs/spec-system/验收清单.md`
  - `docs/spec-system/验收清单审查报告.md`

  - `docs/spec-system/hooks验证方案.md`
  - `docs/spec-system/evidence目录规范.md`
- 当前仓库的 Git 状态
- 真实项目的构建、测试、校验入口
- 当前 `feature.json` 模板与示例

## 本阶段必须完成

1. 固化 `required_checks` 结构化 schema，并把工程验证入口接到结构化字段上
   - 每一条检查至少包含：
     - `id`
     - `name`
     - `shell`
     - `command_windows`
     - `command_bash`
     - `cwd`
     - `timeout_sec`
     - `allow_network`
     - `evidence_file`
     - `stop_on_fail`
   - 必须明确 `command_windows` 是权威执行字段，`command_bash` 仅作为非 Windows 兼容字段。
   - hooks、自动验证、CI 对检查项的读取和判断，必须基于结构化字段，而不是自由文本或字符串模糊匹配。
   - `required_checks`、实际执行命令、CI 结果、evidence 文件之间必须建立可追溯映射。

2. 固化 `feature.json` 状态机与 `done / blocked` 工程闭环
   - 明确固定状态顺序：`draft -> review -> approved -> implementing -> verifying -> done | blocked`
   - 非法跳转必须被拒绝并留下原因记录。
   - 必须明确各状态推进责任，至少覆盖：
     - `approved -> implementing`
     - `implementing -> verifying`
     - `verifying -> done`
     - `verifying -> blocked`
   - `verifying -> done` 只能在全部必需检查与声明的 CI 检查通过后推进。
   - `verifying -> blocked` 必须成为正式出口，而不是口头结论或临时备注。

3. 补齐 Windows 优先执行策略
   - 默认优先 Node.js 脚本，不把 bash-only 路径设计为关键执行路径。
   - 必要 shell 必须显式声明 `shell: powershell`。
   - 所有关键检查与验证入口，必须先满足 Windows 可运行。
   - `required_checks` 与 CI 接入文档必须以 Windows 为第一验证平台。

4. 建立 Git 分支、提交与回滚治理
   - 明确 `feature/<feature-id>` 分支命名。
   - 进入 `implementing` 前，必须验证当前工作分支与 `feature-id` 一致。
   - 明确最小提交要求：至少包含 `feature-id`、当前阶段、变更摘要。
   - 明确失败后的回滚策略，默认优先可审计的 Git 回滚方式。

5. 建立最小 CI 闭环，并把 `done` 与 CI 证据绑定
   - 至少纳入：
     - `lint`
     - `typecheck`
     - `unit test`
     - 必要的 `smoke`
   - 不允许只凭本地成功推进到 `done`。
   - 必须把 CI 结果纳入 evidence 链或可追溯记录。
   - 必须明确哪些 CI 检查属于声明的必需检查，避免“本地通过 / CI 未声明”分叉。

6. 明确真实可运行示例的工程化归属
   - 不能继续让 `example-login` 的后续命运悬空。
   - 本阶段必须基于真实工程环境作出二选一处理：
     - 明确在本阶段把占位示例升级为真实可运行示例，或指定等价真实 feature 作为工程演练样例
     - 明确在本阶段依据真实工程环境将该事项判定为 `blocked`，写清阻塞原因、所缺条件和后续另立项前提
   - 该事项不得再被模糊推给 05。

7. 为 05 提供稳定输入基线
   - 05 只允许在本阶段已经稳定的 `required_checks`、状态机、Git、CI、Windows 基线上做自动化增强。
   - 05 不得回头补 04 应完成的工程基线缺口。

## 允许修改范围

只允许修改：

- `docs/spec-system/**`
- `.claude/rules/*.md`
- `.claude/hooks/*`
- `features/**`
- `tests/**`
- Git / CI 相关配置目录

## 禁止事项

- 禁止引入 deploy / release / CD 落地
- 禁止实现复杂 fix-loop
- 禁止扩展完整第二阶段 agents / subagents 编排
- 禁止把 command 兼容层重新做成主入口
- 禁止伪造 CI 结果、伪造 evidence、伪造真实示例
- 禁止在缺少真实工程环境时继续把本阶段标记为完成

## 本阶段产出

至少应产出：

1. `required_checks` 结构化 schema 与字段权威说明
2. `feature.json` 状态推进、`done / blocked` 闭环与责任边界说明
3. Windows 优先执行策略与对应检查入口说明
4. Git 分支、提交、回滚治理文档
5. 最小 CI 配置或接入方案
6. 至少一条真实 CI 证据样例
7. 对真实可运行示例的明确处理结论：
   - 升级方案，或
   - `blocked` 结论与另立项前提

## 完成判定

以下条件同时满足才算完成：

1. `required_checks` 已结构化，且字段定义足以直接驱动 hook / 自动验证 / CI。
2. `command_windows` 的权威地位已明确，关键路径不存在 bash-only 依赖。
3. `feature.json` 的状态机、非法跳转拒绝、`done / blocked` 正式闭环已经文档化并可执行。
4. 至少一条真实 CI 流程存在并可追溯到 evidence。
5. `done` 已与“全部必需检查 + 声明 CI 检查通过”绑定。
6. 真实可运行示例的归属已经明确，不再悬空。
7. 05 继续推进时，不需要再回头补 04 的工程基线。

## 需要人工介入的点

1. 第二阶段开始前的 phase gate 批准
2. CI 平台接入所需环境变量、密钥、仓库权限
3. 若需把占位示例升级为真实工程演练，需确认目标仓库、资源和验收口径
4. 若真实工程环境缺失，需人工确认 `blocked` 与后续另立项范围
