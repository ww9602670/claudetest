# 阶段需求 04：第二阶段工程化接入

## 阶段目标

把治理体系从“本地可演示”提升到“工程团队可执行”，重点补上 Git 和 CI 两条工程事实链。

本阶段属于第二阶段开始，必须先获得明确的 phase gate 批准。

## 前置条件

必须同时满足：

1. 已完成 `03-第一阶段可审闭环增强.md`
2. `feature.json` 中已经有明确的阶段闸门批准信息
3. 当前仓库或目标仓库已经具备真实可运行的工程环境

若没有真实可运行工程环境，本阶段应直接标记为 blocked，而不是伪造 CI 闭环。

## 本阶段输入

只读取以下内容：

- 第一阶段收口后的规则、hooks、skills
- `docs/需求.md` 中 Git / CI 相关章节
- 当前仓库的 Git 状态
- 真实项目的构建、测试、校验入口

## 本阶段必须完成

1. 建立 Git 分支治理
   - 明确 `feature/<feature-id>` 分支命名
   - 明确进入 implementing 前的分支校验规则
   - 明确提交信息最小要求

2. 建立最小 CI 闭环
   - 至少覆盖：
     - `lint`
     - `typecheck`
     - `unit test`
     - 必要 `smoke`

3. 将 `done` 与 CI 证据绑定
   - 不允许只凭本地执行成功进入 `done`
   - 必须能追溯到一条明确 CI 结果

4. 将 checks 与 evidence 链接起来
   - 检查项、命令、CI 结果、证据文件之间要能对应

## 允许修改范围

只允许修改：

- `docs/spec-system/**`
- `.claude/rules/*.md`
- `.claude/hooks/*`
- `features/**`
- Git / CI 相关配置目录

## 禁止事项

- 禁止引入 deploy / release
- 禁止实现复杂 fix-loop
- 禁止一次性扩展完整 subagents 体系
- 禁止伪造 CI 结果或占位通过记录

## 本阶段产出

至少应产出：

1. Git 分支与提交治理文档
2. 最小 CI 配置或接入方案
3. 一条真实的 CI 证据样例
4. `done` 状态与 CI 结果的绑定规则

## 完成判定

以下条件同时满足才算完成：

1. 至少一条真实 feature 分支命名规则已经可执行
2. 至少一条真实 CI 流程存在并可追溯
3. `done` 不再只是本地口头结论
4. 检查结果与证据链之间存在明确映射

## 需要人工介入的点

1. 第二阶段开始前的 phase gate 批准
2. CI 平台接入所需环境变量、密钥或仓库权限
