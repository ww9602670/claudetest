# 阶段需求 01：现状收敛与口径统一

## 阶段目标

本阶段只做一件事：  
把当前仓库中“需求文档、验收清单、规则实现、示例定位”之间的分叉点收口，形成单一执行口径。

本阶段**不追求新增能力**，只追求后续阶段不再建立在歧义之上。

## 本阶段输入

只读取以下文件：

- `docs/需求.md`
- `docs/spec-system/验收清单.md`
- `docs/spec-system/验收清单审查报告.md`
- `CLAUDE.md`
- `.claude/rules/*.md`
- `.claude/hooks/*.js`
- `docs/spec-system/模板/AI执行版/feature.json`
- `features/example-login/*`

除上述文件外，不要主动扩展阅读范围。

## 本阶段必须解决的口径问题

1. `/project:*` 的验收口径
   - 是“仓内任何字面都不能出现”
   - 还是“不得继续作为实际命令入口”

2. hooks 注册的正式事实源
   - 是项目级 `.claude/settings.json`
   - 还是用户级 `~/.claude/settings.json`
   - 如运行时必须依赖用户级配置，需明确它是“环境接入条件”，不是“项目治理事实源”

3. 批准字段命名统一
   - `approved_note` / `approval_note`
   - `approved_method` / `approval_method`

4. `example-login` 的定位统一
   - 真实可运行示例
   - 或占位演示示例

5. 第一阶段验收是否允许“占位示例”
   - 若允许，必须在验收清单中明确写出
   - 若不允许，后续阶段必须改为真实示例

## 允许修改范围

只允许修改以下内容：

- `docs/spec-system/**`
- `docs/需求.md`
- `CLAUDE.md`
- `.claude/rules/*.md`
- `docs/spec-system/模板/AI执行版/feature.json`
- `features/example-login/*.md`
- `features/example-login/feature.json`

## 禁止事项

- 禁止修改任何业务代码
- 禁止新增 CI、Git、deploy、release 相关能力
- 禁止扩展 agents / subagents 体系
- 禁止开始修 hook 逻辑漏洞
- 禁止补做第二阶段能力

## 本阶段产出

必须产出一组清晰、可执行的统一结论，至少覆盖：

1. 第一阶段的正式验收口径
2. hooks 注册的正式归属
3. `feature.json` 批准字段的唯一命名
4. `example-login` 的正式定位
5. 后续阶段允许依赖的唯一基线

建议以一份短文档或对现有文档的直接修订完成，不要再生成新的长总报告。

## 完成判定

满足以下条件才算完成：

1. 以上 5 个口径问题都有明确结论
2. 结论已经落到仓库文件中，而不是停留在会话口头说明
3. 后续阶段文档可以只基于这些结论继续推进，不再需要二次猜测

## 需要人工介入的点

以下问题必须由人决定，模型不能擅自定案：

1. `/project:*` 采用严格字面禁用，还是仅禁止作为执行入口
2. `example-login` 是否必须升级为真实示例
3. hooks 注册是否接受“项目内文档 + 用户级实际配置”的双层方案
