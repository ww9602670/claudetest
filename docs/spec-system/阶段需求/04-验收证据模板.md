# 阶段 04 验收证据模板（修订版）

用途：用于判定 04 是否“真实完成”，防止仅凭文档结论判定通过。

---

## 一、证据总原则

1. 不允许伪造证据
2. 不允许用占位文本替代真实执行记录
3. 必需证据缺失时，不得判定 04 完成
4. 所有证据必须可追溯到具体任务项与时间

---

## 二、证据存放位置与命名

### 2.1 存放位置

- feature 级证据：`features/<feature-id>/evidence/`
- 04 阶段汇总证据（建议）：`docs/spec-system/阶段需求/evidence/04/`

### 2.2 命名规则

统一命名：

`04-<任务编号>-<证据类别>-<YYYYMMDD>-<HHMMSS>.<ext>`

示例：

- `04-T3-schema-check-20260329-153000.md`
- `04-T7-ci-run-20260329-160500.txt`
- `04-T8-real-feature-proof-20260329-162000.md`

---

## 三、证据分类与必需性

| 验收项 | 必需/补充 | 必需证据内容 | 缺失后果 |
| --- | --- | --- | --- |
| required_checks schema 对齐 | 必需 | 字段完整性核对记录（含字段清单） | 不得判定完成 |
| phase2 启动真实性 | 必需 | `feature.json` 关键字段快照与判定记录 | 不得启动实施 |
| 真实工程环境最小成立 | 必需 | 真实命令入口与一次真实执行输出 | 不得启动实施 |
| Git 分支治理 | 必需 | 分支命名、提交信息、回滚策略证据 | 不得判定工程化闭环 |
| CI 最小闭环 | 必需 | lint/typecheck/unit/smoke（适配项）结果 | 不得进入 done |
| CI 与 evidence 绑定 | 必需 | `required_checks.id -> CI结果 -> evidence_file` 映射表 | 不得进入 done |
| 真实示例归属结论 | 必需 | 升级或 blocked 的明确结论文档 | 不得通过 04 验收 |
| 04→05 交接判定 | 必需 | 交接条件逐项核对表 | 不得进入 05 |
| 辅助说明截图/日志 | 补充 | 运行截图、平台日志、补充说明 | 可补充但不可替代必需证据 |

## 五、否决条件（任一命中即不通过）

1. 必需证据任一缺失
2. 证据无法追溯到具体任务项
3. 仅有口头结论，无文件化记录
4. 仍以 demo_only 作为 phase2 启动依据
