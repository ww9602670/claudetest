# 阶段需求 05：第二阶段自动化增强

## 阶段目标

在工程化接入已经稳定的前提下，补充更强的自动执行、自动停止、自动汇总能力。

本阶段只做“增强”，不再回头承担第一阶段补洞职责。

## 前置条件

必须先完成：

- `docs/spec-system/阶段需求/04-第二阶段工程化接入.md`

如果 Git 和 CI 还不稳定，本阶段不得开始。

## 本阶段输入

只读取以下内容：

- 已稳定的 hooks、rules、skills、agents
- 已稳定的 Git / CI 接入方案
- `docs/需求.md` 中 Stop、fix-loop、subagents、自动化增强相关章节

## 本阶段必须完成

1. 实现停止保护
   - `Stop`
   - `SubagentStop`
   - `override_stop_guard`
   - `max_stop_blocks`

2. 实现更完整的 fix-loop
   - 明确轮次上限
   - 明确重复失败停止
   - 明确人工接管升级条件

3. 强化 agents / subagents 编排
   - 明确职责
   - 明确自动委派边界
   - 明确工具范围和输出物

4. 强化证据汇总
   - 自动生成更稳定的检查摘要
   - 自动汇总多轮迭代记录

5. 视需要补入 command 兼容层
   - 只能作为兼容层
   - 不能重新成为主入口

## 允许修改范围

允许修改：

- `.claude/hooks/*`
- `.claude/skills/**`
- `.claude/agents/*`
- `docs/spec-system/**`
- `features/**`
- 命令兼容层相关文件

## 禁止事项

- 禁止默认引入 deploy / release
- 禁止跳过人工 phase gate
- 禁止把高风险自动化能力直接设为默认自动触发
- 禁止破坏前一阶段已验证通过的 Git / CI 基线

## 本阶段产出

至少应产出：

1. Stop 保护与逃生口实现
2. 更完整的 fix-loop 规则与脚本
3. 更完整的 agents / subagents 文档
4. 更强的证据汇总能力
5. 如需要，最小 command 兼容层

## 完成判定

以下条件同时满足才算完成：

1. 自动验证和自动修复可以安全停止
2. 达到边界条件时能够稳定升级为人工接管
3. 增强能力不会破坏已有审计性和工程可追溯性
4. 高风险能力仍然需要显式人工触发或批准

## 需要人工介入的点

1. 所有高风险自动化能力的启用批准
2. 是否允许引入兼容 command 层
3. 是否允许继续向 deploy / release 扩展

## 本阶段之后

本阶段完成后，如要继续新增：

- deploy
- release
- 更高权限自动操作

应单独立项，不再复用本阶段文档直接扩展。
