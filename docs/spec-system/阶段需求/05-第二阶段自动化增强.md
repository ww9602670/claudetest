# 阶段需求 05：第二阶段自动化增强

## 阶段目标

在 04 的工程化基线上，补齐真正可用的自动化控制流与自动化治理能力。本阶段不仅覆盖 `Stop` / fix-loop / agents / evidence 汇总，还必须显式承接以下自动化增强缺口：

- `UserPromptSubmit`
- Stop 逃生口
- fix-loop 完整边界保护
- command / skill 治理
- 背景知识类与目录专用 skill 治理
- agents / subagents 完整职责体系
- “正式验证清单”作为最终总验收任务的安排

本阶段只做“自动化增强”，不回头补 04 的 `required_checks`、状态机、Windows shell、Git、CI 基线。

## 前置条件

必须先完成：

- `docs/spec-system/阶段需求/04-第二阶段工程化接入.md`

并且同时满足：

1. `required_checks` schema 已稳定，且已作为验证入口的唯一结构化来源。
2. `feature.json` 状态机、`done / blocked` 闭环已稳定。
3. Windows 优先执行路径与 Git / CI 基线已稳定。
4. 04 已对真实可运行示例给出明确结论：已升级，或已正式 `blocked`。

如果 04 仍存在工程基线缺口，或真实工程环境仍未满足，本阶段不得开始。

## 本阶段输入

只读取以下内容：

- 已稳定的 hooks、rules、skills、agents
- 已稳定的 `required_checks`、状态机、Git / CI、Windows 执行方案
- `docs/需求.md` 中 `UserPromptSubmit`、Stop、fix-loop、command / skill、agents / subagents、正式验证清单相关章节
- `docs/spec-system/hooks验证方案.md`
- `docs/spec-system/evidence目录规范.md`
- `docs/spec-system/验收清单.md`

## 本阶段必须完成

1. 引入 `UserPromptSubmit` 控制点
   - 检查用户请求是否超出当前阶段或当前 feature 状态允许范围。
   - 必要时提供提示型处理。
   - 必要时提供阻断型处理。
   - 必须能提示当前 feature 的前置条件缺失、状态不允许或阶段不匹配。

2. 实现 Stop / SubagentStop 与逃生口闭环
   - 必须覆盖：
     - `Stop`
     - `SubagentStop`
     - `override_stop_guard`
     - `max_stop_blocks`
   - 当停止条件未满足时，必须阻断停止并要求继续。
   - 当连续阻断超过阈值时，必须自动降级为人工接管。
   - 每次 Stop 逃生口触发都必须记录原因与时间。

3. 补全 fix-loop 完整边界保护
   - 必须明确：
     - `max_fix_rounds`
     - 重复失败停止
     - 新增失败数上升时停止
     - 人工接管升级条件
     - 缺少可执行验证命令时停止
   - 自动验证循环中的命令，只能来自 04 已稳定的 `required_checks` 结构化白名单。
   - 不允许靠自由文本猜测“下一条该跑什么命令”。

4. 固化 command / skill 治理
   - 禁止同名 skill 与 command 并存。
   - 必须提供显式 command → skill 映射表。
   - command 只能作为兼容层，不得重新成为主入口。
   - 若需要兼容层，只允许最小化补入，不得扩张为第二套长期主体系。

5. 固化背景知识类、目录专用、阶段专用 skill 治理
   - 背景知识类、内部参考类、只供其他流程读取的 skill，必须显式设置 `user-invocable: false`。
   - 目录专用、阶段专用、功能专用 skill，必须显式配置 `paths`。
   - 如属重型研究、长上下文、分叉式工作流，必须显式说明是否使用 `context: fork` 或等价隔离策略，以及是否绑定特定 agent。

6. 补齐 agents / subagents 完整职责体系
   - 每类 agent / subagent 至少要明确：
     - 名称
     - 描述
     - 主动使用时机
     - 允许自动委派时机
     - 工具范围
     - 输出产物
     - 最大工作边界
   - 不允许只保留“旧 agent 还在”这种模糊状态。
   - 必须明确哪些 agent 属于阶段必用，哪些只在特定场景下触发。

7. 强化 evidence 汇总，并把正式验证清单安排为最终总验收任务
   - 自动汇总检查摘要、多轮迭代记录、阻塞原因、Stop 逃生口触发记录、委派记录。
   - 必须明确：`docs/需求.md` 中的“正式验证清单”在本阶段之后作为最终总验收任务执行，而不是停留在口头建议。
   - 最终总验收必须与 evidence、CI 结果、`feature.json` 状态记录绑定，不得只看会话输出。

8. 视需要补入最小 command 兼容层
   - 仅在迁移兼容确有必要时补入。
   - 补入时必须受本阶段 command / skill 治理约束。

## 禁止事项

- 禁止默认引入 deploy / release
- 禁止跳过人工 phase gate
- 禁止把高风险自动化能力直接设为默认自动触发
- 禁止破坏 04 已验证通过的 `required_checks`、状态机、Git / CI、Windows 基线
- 禁止回退到自由文本猜命令或口头判定 `done`
- 禁止把 04 尚未解决的真实示例归属继续悬空

## 本阶段产出

至少应产出：

1. `UserPromptSubmit` 控制规则或等价实现方案
2. Stop 保护与逃生口实现
3. 更完整的 fix-loop 规则与脚本
4. command / skill 映射与冲突治理文档
5. 背景知识类与目录专用 skill 的治理规则
6. 更完整的 agents / subagents 文档
7. 更强的 evidence 汇总能力
8. 把正式验证清单接入最终总验收的任务安排
9. 如确有必要，最小 command 兼容层

## 完成判定

以下条件同时满足才算完成：

1. `UserPromptSubmit` 已能按阶段边界和 feature 状态进行提示或阻断。
2. 自动验证和自动修复可以安全停止，且不会因 Stop 保护自身导致无限阻断。
3. 达到重复失败、失败升级、缺少验证命令等边界条件时，能够稳定升级为人工接管。
4. command / skill 不存在同名冲突，兼容层不再与主入口争权。
5. 背景知识类与目录专用 skill 的 `user-invocable`、`paths`、上下文隔离策略已经明确。
6. agents / subagents 的职责、工具范围、输出物和委派边界可审计。
7. 正式验证清单已被明确安排为本阶段之后的最终总验收任务，并与 evidence / CI / `feature.json` 绑定。
8. 本阶段增强能力未破坏 04 已建立的工程可追溯性。

## 需要人工介入的点

1. 所有高风险自动化能力的启用批准
2. 是否允许引入 command 兼容层
3. 若 04 已将真实示例判定为 `blocked`，是否在 05 之后单独立项补足真实工程演练
4. 是否允许继续向 deploy / release 扩展

## 本阶段之后

本阶段完成后，如要继续新增：

- deploy
- release
- 更高权限自动操作
- 在缺少真实工程环境前提下另立项补足真实可运行示例

应单独立项，不再复用本阶段文档直接扩展。
