# 迁移说明：从旧体系到新体系

## 旧体系 → 新体系

| 旧体系 | 新体系 | 说明 |
|--------|--------|------|
| `specs/<功能>/` | `features/<feature-id>/` | 执行态目录已迁移 |
| `/project:spec` | `/spec` | 命令前缀去掉 `/project:` |
| `/project:design` | `/design` | 同上 |
| `/project:tasks` | `/tasks` | 同上 |
| `/project:implement` | `/implement` | 同上 |
| `/project:verify` | `/verify` | 同上 |
| 无状态文件 | `feature.json` | 新增唯一状态源 |
| `spec.md` 单文档 | 小白版 + AI 执行版双文档 | 新增小白版目标说明 |

## 迁移步骤

### 对于新功能

直接使用新体系：

```
/spec <feature-id> <需求描述>
```

功能文件将在 `features/<feature-id>/` 下创建。

### 对于 specs/ 下的已有文档

1. 在 `features/` 下创建对应目录
2. 复制已有的 `spec.md`、`design.md`、`tasks.md`、`verify.md`
3. 补充创建 `feature.json`（参考模板）
4. 补充创建小白版文档（`goal.md`、`plan.md`、`steps.md`、`acceptance.md`）
5. 原 `specs/<功能>/` 目录可保留归档，但不再作为执行态目录

### specs/ 目录的新定位

`specs/` 目录现在仅作为**历史归档**，不再作为执行态目录。

新的执行态目录只有 `features/`。

## 注意事项

- 禁止在 `specs/` 下创建新功能目录
- 禁止继续使用 `/project:*` 命令（已无效）
- 禁止使用 `feature.yaml`，只使用 `feature.json`
- `.claude/commands/` 下的旧命令文件保留为兼容别名，但 `.claude/skills/` 下的同名 skill 优先
