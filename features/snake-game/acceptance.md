# 验收标准 — 网页版贪食蛇游戏

> 小白版 | feature-id: snake-game
> 
> **目的**: 明确什么样的游戏算是"做完了"

---

## 验收标准是什么意思？

验收标准就是用来判断这个游戏是否完成、是否合格的清单。

就像考试一样，我们需要设定一些标准，达到这些标准就算通过。

---

## 两层验收标准

### 第一层：机制验证验收（最重要）

**这层验收是判断整个开发方法是否成功的标准。**

✅ **必须满足**：
1. **有完整的需求文档**
   - 写清楚了游戏需要哪些功能
   - 写清楚了游戏怎么玩
   - 写清楚了什么时候游戏结束

2. **有完整的设计文档**
   - 写清楚了用什么技术做
   - 写清楚了怎么做
   - 写清楚了可能会遇到什么问题

3. **有完整的任务清单**
   - 把工作拆成了小任务
   - 每个任务都有明确的目标
   - 任务之间有清晰的依赖关系

4. **有普通人能看懂的说明**
   - 不懂技术的人也能看懂
   - 说明了游戏是什么
   - 说明了怎么玩
   - 没有用太多专业术语

5. **有开发过程的证据**
   - 记录了做了什么
   - 记录了怎么做的
   - 记录了测试结果
   - 记录了遇到的问题和解决方法

6. **有可执行的检查项**
   - 有方法检查文件是否存在
   - 有方法检查文件是否为空
   - 有方法检查基本功能是否正常

### 第二层：游戏功能验收（次要）

**这层验收是判断游戏本身是否完整好用。**

#### 必须有的功能（P0）
- [ ] 游戏能在浏览器中打开
- [ ] 能看到蛇在移动
- [ ] 能用方向键控制蛇的方向
- [ ] 蛇能吃到食物
- [ ] 吃到食物后得分增加
- [ ] 吃到食物后蛇变长
- [ ] 蛇撞墙后游戏结束
- [ ] 蛇撞到自己后游戏结束
- [ ] 游戏结束后能重新开始

#### 最好有的功能（P1）
- [ ] 能看到当前得分
- [ ] 游戏结束后能看到最终得分
- [ ] 速度会随着得分增加而变快
- [ ] 速度变化明显但不至于无法控制

#### 可以没有的功能（P2）
- [ ] 复杂的视觉效果
- [ ] 音效
- [ ] 排行榜
- [ ] 暂停功能
- [ ] 移动端适配

**重要说明**：如果第一层验收（机制验证）全部通过，但第二层验收中有个别 P1/P2 功能不完美，仍然可以判定任务完成。因为本次任务的核心目标是验证开发方法，而不是做一个完美的游戏。

## 具体验收检查项

### 文档完整性检查

**AI 执行版文档**（给开发人员看的）：
- [ ] `spec.md` - 需求规格文档存在且完整
- [ ] `design.md` - 技术设计文档存在且完整
- [ ] `tasks.md` - 任务清单存在且完整
- [ ] `verify.md` - 验证计划存在且完整

**小白版文档**（给非技术人员看的）：
- [ ] `goal.md` - 目标说明存在且完整
- [ ] `plan.md` - 实施计划存在且完整
- [ ] `steps.md` - 操作步骤存在且完整
- [ ] `acceptance.md` - 验收标准存在且完整

**项目说明**：
- [ ] `README.md` - 项目说明存在且清晰

### 文档可读性检查

**非技术人员检查**：
- [ ] 找一个不懂技术的人，让他看小白版文档
- [ ] 他能看懂这个游戏是干什么的吗？
- [ ] 他能看懂怎么操作吗？
- [ ] 他能看懂游戏规则吗？
- [ ] 他觉得文档容易理解吗？

### 游戏功能检查

**打开游戏**：
- [ ] 双击 `index.html` 能在浏览器中打开
- [ ] 页面显示正常，没有乱码
- [ ] 能看到游戏画面

**基本功能**：
- [ ] 能看到蛇在移动
- [ ] 按方向键能改变蛇的移动方向
- [ ] 蛇不会直接掉头（比如从上直接往下）
- [ ] 蛇能吃到红色的食物
- [ ] 吃到食物后得分增加 10 分
- [ ] 吃到食物后蛇的身体变长一节

**游戏结束**：
- [ ] 蛇撞到墙壁时游戏结束
- [ ] 蛇撞到自己时游戏结束
- [ ] 游戏结束时显示"游戏结束"文字
- [ ] 游戏结束时显示最终得分

**重新开始**：
- [ ] 游戏结束后有"重新开始"按钮
- [ ] 点击按钮后游戏重置
- [ ] 得分清零
- [ ] 蛇回到初始位置和长度

### 速度控制检查（P1 功能）

- [ ] 游戏开始时速度适中（不太快也不太慢）
- [ ] 吃到几个食物后速度明显变快
- [ ] 速度不会快到无法控制

### 性能检查

- [ ] 页面加载时间小于 2 秒
- [ ] 按键反应灵敏（没有明显延迟）
- [ ] 游戏画面流畅（不卡顿）

### 浏览器兼容性检查

在以下浏览器中测试：
- [ ] Chrome 浏览器能正常玩
- [ ] Firefox 浏览器能正常玩
- [ ] Edge 浏览器能正常玩

### 证据完整性检查

- [ ] `evidence/` 目录存在
- [ ] 有开发过程的记录
- [ ] 有测试结果的记录
- [ ] 有状态变更的记录
- [ ] 如果有调整，有调整说明的记录

### 检查项执行记录

- [ ] 所有定义的 required_checks 都执行过
- [ ] 每个检查都有对应的证据文件
- [ ] 证据文件内容完整、真实

## 验收流程

### 第一步：自检
开发者按照上面的清单自己检查一遍，记录结果。

### 第二步：非技术用户测试
找一个不懂技术的人来测试：
1. 让他看小白版文档，看能不能理解
2. 让他按照步骤打开游戏
3. 让他玩一局游戏
4. 问他有没有问题，能不能理解

### 第三步：收集反馈
记录测试中发现的问题和建议。

### 第四步：判定
根据收集的结果判定：
- ✅ **通过** - 机制验证全部完成，游戏基本功能正常
- ⚠️ **有条件通过** - 机制验证完成，游戏有小问题但不影响验证
- ❌ **不通过** - 机制验证不完整或有严重问题

---

## 通过标准

### ✅ 完全通过（理想状态）
- 所有文档完整且清晰
- 所有 P0 功能正常
- 大部分 P1 功能正常
- 证据记录完整
- 非技术人员能理解

### ⚠️ 有条件通过（可接受）
- 文档完整但有小瑕疵
- 所有 P0 功能正常
- 部分 P1 功能有小问题
- 证据记录基本完整
- 非技术人员能基本理解

**什么情况下可以判定为有条件通过**：
- 游戏能正常玩，但视觉效果不够美观
- 速度控制功能有小问题，但不影响基本游戏
- 文档有个别地方不够清楚，但不影响理解

### ❌ 不通过（需要改进）
- 文档不完整
- P0 核心功能有严重问题
- 机制验证不完整
- 证据记录缺失

---

## 不通过时怎么办？

### 问题1：文档不完整
**解决**：补充缺失的文档内容

### 问题2：核心功能有问题
**解决**：
- 如果是简单问题，直接修复
- 如果是复杂问题，记录到 evidence，可能需要人工介入

### 问题3：非技术人员看不懂
**解决**：改写小白版文档，用更通俗的语言

### 问题4：证据不完整
**解决**：补充缺失的 evidence 记录

---

## 验收后的状态

### 如果验收通过
任务可以标记为 `done`，本次 04 任务完成。

### 如果验收有条件通过
任务可以标记为 `done`，但要在 evidence 中记录待改进项。

### 如果验收不通过
任务继续改进，直到达到有条件通过的标准。

---

## 最终判断标准

**记住**：本次任务最重要的目标是验证机制，而不是做出一个完美的游戏。

只要满足：
- ✅ 按照需求→设计→任务的流程做出来了
- ✅ 过程中有清晰的记录
- ✅ 产出了普通人能看懂的文档
- ✅ 游戏基本能玩

就可以判定为完成！

游戏功能上的一些小问题，可以记录到 evidence 中，作为后续迭代的内容。
