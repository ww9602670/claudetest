# 任务拆解（Tasks）生成

基于 Spec 和 Design 文档，拆解为可逐条执行的任务清单。

## 执行步骤

1. 读取 `specs/<功能名>/spec.md` 和 `specs/<功能名>/design.md`（功能名从 $ARGUMENTS 获取）
2. 如果 design.md 不存在，提示用户先执行 `/project:design`
3. 在同一目录下创建 `tasks.md`

## tasks.md 文档结构（严格遵循）

```markdown
# <功能名称> - 任务清单

> 基于 [spec.md](./spec.md) 和 [design.md](./design.md) 生成
> 生成时间：YYYY-MM-DD

## 执行规则

- 按顺序逐条执行，不得跳过
- 每完成一条将 `[ ]` 改为 `[x]` 并记录完成时间
- 遇到阻塞时记录原因，继续下一条可执行的任务
- 每个任务完成后进行自验证

## 任务列表

### 阶段一：基础搭建

- [ ] **Task 1.1**：<任务标题>
  - 目标：明确描述这个任务要达成什么
  - 操作：
    1. 具体步骤1
    2. 具体步骤2
  - 涉及文件：`path/to/file`
  - 验证方式：如何确认这个任务完成了
  - 预计改动量：S/M/L

- [ ] **Task 1.2**：<任务标题>
  ...

### 阶段二：核心实现

- [ ] **Task 2.1**：<任务标题>
  ...

### 阶段三：测试与收尾

- [ ] **Task 3.1**：<任务标题>
  ...

## 依赖关系

- Task 2.1 依赖 Task 1.1 和 Task 1.2
- （列出所有任务间依赖）

## 总计

- 总任务数：N
- 预计总改动量：S/M/L
```

## 注意事项

- 每个 Task 必须足够小，能在一次交互中完成
- 每个 Task 必须有明确的验证方式
- 涉及文件必须精确到路径
- 阶段划分应遵循：基础→核心→测试的顺序
- 完成后告知用户文件路径，并提示下一步执行 `/project:implement <功能名>`
