# 需求规格（Spec）生成

根据用户输入的需求描述，生成结构化的需求规格文档。

## 执行步骤

1. 分析用户提供的需求描述（即 $ARGUMENTS）
2. 如果需求不够清晰，**主动向用户提问**以澄清关键细节，不要假设
3. 确认需求理解无误后，在 `specs/<功能名>/` 目录下创建 `spec.md`

## spec.md 文档结构（严格遵循）

```markdown
# <功能名称>

## 1. 概述
简要描述该功能的目的和价值（2-3句话）

## 2. 用户故事
- 作为 <角色>，我希望 <行为>，以便 <价值>
- （列出所有相关用户故事）

## 3. 功能需求
### 3.1 核心需求
- [ ] 需求1
- [ ] 需求2

### 3.2 次要需求
- [ ] 需求1

## 4. 非功能需求
- 性能：
- 安全性：
- 兼容性：

## 5. 验收标准
- [ ] 标准1：具体的、可验证的条件
- [ ] 标准2

## 6. 边界情况与异常处理
| 场景 | 预期行为 |
|------|----------|
| 场景1 | 处理方式 |

## 7. 依赖与约束
- 外部依赖：
- 技术约束：
- 时间约束：

## 8. 开放问题
- （记录尚未确定的问题，需要用户进一步确认）
```

## 注意事项

- 功能目录名使用英文 kebab-case
- 文档内容使用中文
- 验收标准必须是具体的、可测试的
- 如果用户没有提供足够信息，在「开放问题」中列出并主动询问
- 完成后告知用户文件路径，并提示下一步执行 `/project:design`
