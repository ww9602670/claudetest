# 技术设计（Design）生成

基于已有的 Spec 文档，生成技术设计方案。

## 执行步骤

1. 找到并读取 `specs/<功能名>/spec.md`（功能名从 $ARGUMENTS 获取，若未指定则列出现有 specs 目录供选择）
2. 分析项目现有代码结构和技术栈
3. 在同一目录下创建 `design.md`

## design.md 文档结构（严格遵循）

```markdown
# <功能名称> - 技术设计

## 1. 技术方案概述
用 2-3 段话描述整体技术方案的思路和选型理由

## 2. 系统架构
### 2.1 架构图（文字描述）
描述模块间的关系和数据流向

### 2.2 核心模块
| 模块 | 职责 | 输入 | 输出 |
|------|------|------|------|

## 3. 数据模型
### 3.1 数据结构
（使用代码块展示关键数据结构/类型定义）

### 3.2 数据流
描述数据从输入到输出的完整流转路径

## 4. 接口设计
### 4.1 对外接口
（API 定义、函数签名、参数说明）

### 4.2 内部接口
（模块间调用关系）

## 5. 文件变更清单
| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 新增 | `src/xxx.ts` | 用途说明 |
| 修改 | `src/yyy.ts` | 改动内容 |

## 6. 技术风险
| 风险 | 影响 | 缓解措施 |
|------|------|----------|

## 7. 测试策略
- 单元测试：
- 集成测试：
- 端到端测试：

## 8. 对 Spec 的反馈
（如果在设计过程中发现 Spec 有遗漏或不合理之处，在此记录）
```

## 注意事项

- 设计必须与 Spec 中的需求一一对应
- 文件变更清单要精确到具体文件路径
- 如果项目已有代码，先阅读相关文件再设计
- 避免过度设计，方案应与需求规模匹配
- 完成后告知用户文件路径，并提示下一步执行 `/project:tasks <功能名>`
